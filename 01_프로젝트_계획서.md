## 0. 1차 프로젝트(HMS) 대비 진화 포인트 (2차 프로젝트의 의미)

| 영역 | 1차 (HMS) | 2차 (QT-AI) | 학습 효과 | 수업 진도 |
| --- | --- | --- | --- | --- |
| 아키텍처 | Spring Boot 모놀리식 | **MSA (Gateway + 6 Services)** | 서비스 경계, 도메인 분리 설계 | ✅ 6.MSA chap01·DDD |
| 프론트엔드 | Mustache SSR (웹) | **Flutter (모바일 크로스플랫폼)** | 모바일 UX, 상태 관리 | ✅ 플러터 chap04~12 |
| 메시징 | 동기 호출만 | **Kafka 이벤트 기반 + 동기 혼합** | 비동기 설계, 이벤트 소싱 | ✅ 6.MSA chap04 (Kafka) |
| 인프라 | Docker Compose 단일 | **Kubernetes (Minikube) + Helm** | 오케스트레이션, 헬름 차트 | ✅ 6.MSA ex08 + chap03 |
| 관측성 | 단순 로그 | **분산 트레이싱 (Jaeger) + 로그 집계 (Loki)** | MSA 운영 디버깅 | ⚠️ 부분 — 직접 학습 필요 |
| AI | RAG 단답형 (ChromaDB) | **코칭형 AI (소크라테스식 꼬리 질문) + 큐티 A~D형 학습** | 프롬프트 엔지니어링 고도화 | — 1차 RAG 경험 활용 |
| 실시간 | 없음 | **AI 응답 SSE 스트리밍 + 알림 WebSocket** | 실시간 통신 패턴 | ✅ 6.MSA chap05 + 실시간통신 |
| 클린 아키텍처 | Layered (전통적) | **UseCase 패턴 + Hexagonal** | 의존성 역전, 포트/어댑터 | ✅ 6.MSA chap03 (UseCase) |
| 데이터 | 단일 DB | **다중 JOIN 다국어 병기 + 학술 자료 통합** | DTO 어그리게이션, 병렬 호출 | — 1차 JPA 경험 활용 |
| 보상 트랜잭션 | 없음 (단일 DB) | **Saga 패턴 + 보상 트랜잭션** | 분산 트랜잭션 이해 | ✅ 6.MSA 보상 트랜잭션 |

> **고차원의 정의:** 인프라 복잡도 ↑ + 도메인 깊이 ↑ + UX 정교함 ↑ 의 3축 동시 진화. 단, 1차에서 발생한 3대 실패 (트랜잭션 누락 / Spring Boot 메서드 환각 / 평문 시크릿 커밋)는 MSA에서 N배 증폭되므로 § 9·§ 10에서 가드레일로 박제.
>
> **수업 진도 반영:** 핵심 스택 대부분이 수업(노션 기술 블로그 「6. MSA」 + 「플러터」)에서 이미 다뤄졌음. 학습곡선 리스크는 **분산 운영(관측성·통합 디버깅)** 영역에 집중.

## 2. 핵심 기능 상세 정의

### 2.1 기능 1 — 입체적 묵상 화면 (Bible + Commentary Sync View)

- **대상:** 모든 로그인 사용자
- **화면 흐름:** 홈 → 성경 선택 (창세기 1:1) → 입체 화면 (한/영 병기 + 주석/논문 패널)
- **사용자 플로우:**
  1. Flutter 클라이언트가 BFF Aggregator에 `GET /api/v1/passages/{book}/{chapter}/{verse}` 호출
  2. BFF가 Bible Service에 한글·영어·주석을 **CompletableFuture로 병렬** 호출
  3. 단일 응답 DTO (`PassageAggregateResponse`)로 묶어 반환
  4. Flutter는 `CustomScrollView` + `Sliver` + `ScrollController` 동기화로 한/영 Sync Scroll 구현 (수업 「Sliver 위젯 완전 정복」 참고)
- **외부 API 호출 설계:**
  - Bible Service 내부: MySQL `KR_BIBLE` / `EN_BIBLE` / `COMMENTARY` 동일 인덱스 JOIN
  - 타임아웃: 각 호출 800ms → 1회 재시도 → 폴백(부분 응답)
  - 캐시: Redis (key: `passage:{book}:{ch}:{v}`, TTL 24h, 핫셋만)
- **데이터 저장:** 사용자 조회 이력은 Kafka `user.activity.tracked` 이벤트 → Journal Service에서 비동기 적재
- **포트폴리오 어필 포인트:** **DTO 어그리게이션 + 병렬 호출 + Sync Scroll** — "왜 BFF 패턴이 필요한가" 면접 단골

### 2.2 기능 2 — AI 코칭형 묵상 대화 (큐티 A~D형 + 소크라테스식, SSE 스트리밍)

- **대상:** 인증된 사용자
- **화면 흐름:** 묵상 대상 구절 선택 → AI 대화창 → 큐티 단계 자동 진행 (A: 관찰 → B: 해석 → C: 적용 → D: 결단)
- **시스템 프롬프트 / 비즈니스 룰:** (강상민 — AI/RAG 도메인 owner)
  - 단계별 시스템 프롬프트 4종 (`PromptTemplate.A_OBSERVATION` 등)
  - 사용자가 답변하면 ChromaDB에서 관련 주석·논문 임베딩 RAG 조회
  - 다음 단계로 넘어가는 조건: "사용자 답변에 핵심 키워드 N개 포함" 또는 "소크라테스식 재질문 3회 후 자동 진행"
  - **신학적 오류 방지 가드:** RAG 컨텍스트 외 추론 금지 + 출처 인용 강제
- **외부 API 호출 설계:**
  - Anthropic Claude API (**Streaming = SSE로 Flutter까지 패스스루**, 수업 「실시간 통신 비교 및 실습」 참고)
  - 타임아웃: 첫 토큰 5s → 폴백("잠시 후 다시 시도해 주세요")
  - 키 관리: K8s Secret, ExternalSecrets 사용 (**평문 커밋 절대 금지** — § 10 가드레일 참조)
- **데이터 저장:** 대화 종료 시 Kafka `ai.session.completed` 이벤트 → Journal Service가 묵상 노트로 저장
- **포트폴리오 어필 포인트:** **소크라테스식 코칭 + RAG + SSE 스트리밍** 결합 — 단순 챗봇 위에 한 단계 위 AI 적용 어필

### 2.3 기능 3 — 묵상 기록 (Journal) & 통계 + 이벤트 소싱

- **대상:** 인증된 사용자
- **화면 흐름:** 마이페이지 → 묵상 기록 리스트 → 상세 (구절·날짜·AI 대화·개인 메모)
- **이벤트 소싱:** Kafka `journal.created`, `journal.updated`, `journal.deleted` → 이벤트 스트림으로 상태 재구성 가능 (수업 「저널링」 참고)
- **데이터 저장:** Journal Service의 MySQL `JOURNAL` + `JOURNAL_EVENT` 테이블
- **포트폴리오 어필 포인트:** **이벤트 소싱 패턴 + Kafka 컨슈머 멱등성**

### 2.4 기능 4 — 인증·회원 (Auth/User)

- JWT 발급 + Refresh Token + 소셜 로그인(Google) 1종
- Gateway에서 JWT 검증 (RS256), 만료 시 401 → Refresh 자동
- 회원 탈퇴는 Soft Delete + 이벤트 발행 (`user.deactivated`)

### 2.5 기능 5 — 실시간 알림 (WebSocket) + 관리자 대시보드

- **알림:** Kafka `notification.requested` → Notification Aggregator → WebSocket으로 Flutter 푸시 (수업 「chap05 - 실시간 알림: WebSocket으로 주문 완료를 즉시 전달하다」 참고)
- **대시보드:** 총 묵상 횟수, 인기 구절 Top 10, AI 호출량, 일/주/월 활성 사용자
- BFF Aggregator가 각 서비스 통계 API 병렬 호출 → 단일 응답

### 6.5 팀 협업 운영 원칙 (작업 스타일 기반)

> 6명 모두 작업 스타일·강점이 다르다. 각자에게 맞는 운영 원칙을 박제해두어 "잘 하는 사람만 갈리는" 1차의 패턴을 반복하지 않는다.

#### A. Definition of Done (DoD) — 모든 작업에 강제

작업이 끝났다고 선언하기 전에 아래 7가지를 자기 PR에 체크리스트로 박제한다. **"여기까지 했으니 끝"이 아니라 "체크리스트 7개가 모두 ✅"가 끝의 정의.**

```
[ ] 도메인 코드 + 단위 테스트 (커버리지 70%+)
[ ] OpenAPI yaml과 응답 schema 일치
[ ] Kafka 컨슈머 사용 시 멱등성 키 + DLQ 설정
[ ] @Transactional 또는 Saga 보상 트랜잭션 정의
[ ] Secret은 K8s Secret/ExternalSecrets로만 (평문 0건)
[ ] 분산 트레이싱 traceId propagation 동작
[ ] README 또는 ADR에 의사결정 기록
```

#### B. 페이스 모니터링 — 깊이 있는 작업자 보호

- 이지윤(Auth)·김태혁(Bible)처럼 **꼼꼼함이 강점인 영역**은 W1 골격을 빠르게 끝내고 W2 화요일에 1차 페이스 점검 회의 (Lead + 본인). 전체 진척의 50% 이상이면 그대로, 미달이면 페어 또는 범위 조정.
- 마감 직전 1주(W4)에 미완료 항목이 보이면 즉시 다른 owner와 페어 또는 우선순위 가중치로 일부 컷.

#### C. 작업 범위 명시 (특히 분산 패턴 영역)

- 이승욱(Journal+Kafka)처럼 **분산 패턴 디테일이 까다로운 영역**은 작업 시작 전 Lead가 작업 카드에 **DoD + 디테일 항목**을 명시한다 (예: "이벤트 소싱 = ① JOURNAL_EVENT 적재, ② read model 재구성, ③ 멱등성 키, ④ DLQ, ⑤ 재처리 시나리오, ⑥ 통합 테스트"). 6개를 명시했으면 6개가 끝나야 done.
- 모호한 범위 ("이벤트 소싱 구현해줘")는 절대 발행 금지.

#### D. 페어 짝 (강점 보완)

| 막힐 때 페어 짝 (1차 리커리어) | 영역 |
| --- | --- |
| 이지윤 ↔ 강태오 (Lead) | Spring Security·OAuth 표준 결정 |
| 김태혁 ↔ 이승욱 | 다중 JOIN 쿼리 튜닝·인덱스 (이승욱 DB 깊이 백업) |
| 강상민 ↔ 강태오 | RAG 통합·SSE 스트리밍 셋업 |
| 이승욱 ↔ 강태오 | Kafka·Schema Registry·멱등성 |
| 김지민 ↔ 강상민 | AI 대화 Flutter UI ↔ AI Service 계약 |

#### E. AI 에이전트 활용 표준화 (전원)

- 모든 PR이 AI 보조이면 `ai-assisted` 라벨 + § 10.3 환각 체크리스트 필수.
- AI 보조 코드는 **사람 1명 이상 라인 단위 검토** 후 머지 (자기 자신만 검토한 PR은 머지 금지).

## 8. 형상관리 전략

### 8.1 Git 브랜치 전략 (MSA 모노레포)

- 단일 모노레포(monorepo): `2nd-Team-Project` 한 저장소에 service 디렉토리 분리
- `main` — 배포 전용
- `develop` — 개발 중심
- `feature/{service}-{이슈번호}-{짧은설명}` (예: `feature/bible-12-passage-cache`)
- `bugfix/...` / `hotfix/...`

### 8.2 커밋 컨벤션 (Conventional Commits + scope=service)

`{type}({scope}): {subject}` — 예: `feat(bible): add Redis passage cache` / `fix(ai): handle ChromaDB timeout`
type: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `ci`, `style`

### 8.3 PR 규칙

- 400줄 이하 권장, 1000줄 초과 시 분할
- 리뷰 2명 (자기 서비스 동료 1명 + Lead 1명) + CI green 후 squash merge
- AI 보조 PR은 `ai-assisted` 라벨 + 환각 체크리스트 (§ 10.3) 적용
- **공유 레이어 변경은 `shared-change` 라벨 강제** + 영향받는 모든 서비스 소유자 리뷰 필수
- **모든 PR은 § 6.5 A의 DoD 체크리스트 ✅** (자기 서비스 owner 책임)

### 8.4 ADR (Architecture Decision Record) — 의사결정 추적

> MSA는 결정 사항이 많아 추적 부재 시 6명이 서로 다른 가정으로 코드를 짜게 됨. ADR로 박제. **특히 Lead의 AI 보조 의사결정은 반드시 ADR로 박제하여 검증 가능하게.**

- 디렉토리: `docs/adr/`
- 파일명: `NNNN-{slug}.md` (예: `0001-msa-service-boundary.md`, `0002-kafka-vs-rabbitmq.md`)
- 표준 섹션: Context · Decision · Consequences · Alternatives · Status (Proposed/Accepted/Superseded) · **Reviewer** (Proposed 단계에 필수: 이승욱·이지윤 중 1명 이상)
- W0~W1에 작성할 후보: 서비스 경계 / 모노레포 vs 멀티레포 / Kafka vs RabbitMQ / SSE vs WebSocket / Flutter 상태관리(RiverPod) / 공유 레이어 소유권

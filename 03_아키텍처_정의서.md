## 3. 전체 구조

QT-AI v1은 하나의 Spring Boot 백엔드 안에서 도메인을 강하게 분리한다. v1의 목표는 운영 복잡도를 줄이면서도, v2에서 MSA로 분리할 수 있을 만큼 도메인 경계를 미리 고정하는 것이다.

```text
Flutter App / Admin Web
        |
        v
  /api/v1/**  (External HTTP API)
        |
        v
qtai-server (Spring Boot 3.3, Java 21)
        |
        +-- gatewayauth
        +-- bff
        +-- bible
        |   +-- journal
        |   +-- songs
        +-- ai
        +-- simulator
        |
        +-- MySQL 8.0
        +-- External Sources
            +-- GitHub public Bible JSON
            +-- QT range source
            +-- DeepSeek API
```

### 3.1 레이어 원칙

| 레이어 | 책임 |
| --- | --- |
| API/Presentation | `/api/v1/**` 외부 요청 수신, 인증/인가 결과 반영, 요청 DTO 검증 |
| BFF/Application | 화면 단위 UseCase 조립, 여러 도메인 Port 호출, 응답 DTO 구성 |
| Domain | 도메인 규칙, 상태 전이, 핵심 모델 관리 |
| Port/API | 도메인 외부에 공개하는 Java Interface와 DTO |
| Infrastructure | DB, 외부 API, 배치, 파일/클립 저장소, 구현 어댑터 |
| Event | 도메인 변경 알림, 캐시 무효화, 알림 큐잉, 관리자 모니터링 연결 |

### 3.2 외부 API와 내부 인터페이스 분리

외부 공개 API는 Flutter 앱, 관리자 화면, 외부 클라이언트가 호출하는 HTTP API다. 기본 prefix는 `/api/v1`로 통일한다.

내부 도메인 인터페이스는 `qtai-server` 내부에서 도메인 간 호출에 사용하는 Java Interface다. 내부 인터페이스는 HTTP 경로가 아니며, 다른 도메인의 Entity, Service, Repository를 직접 import하지 않는다.

| 구분 | 형태 | 예시 | 관리 문서 |
| --- | --- | --- | --- |
| 외부 공개 API | HTTP | `GET /api/v1/qt/today` | `04_API_명세서.md` |
| 내부 도메인 인터페이스 | Java Interface | `BibleQueryPort`, `ExplanationQueryPort` | 이 문서, `04_API_명세서.md` |
| 내부 이벤트 | Java Event | `AiContentGeneratedEvent` | 이 문서 |

## 4. 도메인 패키지 구조

```text
qtai-server/
└── src/main/java/com/qtai/
    ├── gatewayauth/
    ├── bff/
    ├── bible/
    │   ├── journal/
    │   └── songs/
    ├── ai/
    └── simulator/
```

| 도메인 | 책임 | 외부에 공개 가능한 것 |
| --- | --- | --- |
| `gatewayauth` | JWT 발급·검증, Google OAuth, Rate Limit, 인증/인가 공통 처리 | 인증 결과 DTO, 인증 검증 Port |
| `bff` | 화면 단위 UseCase, 여러 도메인 호출 집계, 외부 API 응답 조립 | 외부 Controller, 화면 응답 DTO |
| `bible` | 성경 JSON 적재, QT 본문 범위 수집, A/B/C 해설 테이블, 묵상 노트·달력(`journal`), 찬양 큐레이션(`songs`), 본문 조회 | `BibleQueryPort`, `ExplanationQueryPort`, `JournalCommandPort`, `JournalQueryPort`, `SongCommandPort`, `SongQueryPort` |
| `ai` | DeepSeek 연동, 해설 사전 생성, 편집자 검증, AI 실행 이력 | `AiRunLogPort`, AI 배치/관리자 트리거 |
| `simulator` | 본문 장면 클립 사전 생성, 상태 저장, 클립 조회 | `SimulatorQueryPort`, `SimulatorClipSavePort` |

`journal`과 `songs`는 최상위 도메인이 아니라 `com.qtai.bible` 내부 하위 모듈로 둔다. 다른 도메인은 `bible`의 내부 Entity나 Service를 직접 import하지 않고 공개 Port만 사용한다.

## 5. 도메인 경계 규칙

### 5.1 허용 규칙

| 호출 방향 | 허용 방식 |
| --- | --- |
| `bff` → `bible` | `BibleQueryPort`, `ExplanationQueryPort` |
| `bff` → `bible/journal` | `JournalCommandPort`, `JournalQueryPort` |
| `bff` → `simulator` | `SimulatorQueryPort` |
| `bff` → `ai` | 관리자 트리거용 Command Port |
| `ai` → `bible` | `ExplanationSavePort`로 C 테이블 저장 |
| `ai` → `simulator` | 시뮬레이터 생성 요청 또는 결과 검증 이벤트 |
| `simulator` → `simulator` | `SimulatorClipSavePort`로 클립·상태 저장 |

### 5.2 금지 규칙

| 금지 항목 | 이유 |
| --- | --- |
| 다른 도메인의 Entity 직접 import | 도메인 모델 결합 방지 |
| 다른 도메인의 Service 직접 import | UseCase 흐름이 도메인 내부 구현에 묶이는 문제 방지 |
| 다른 도메인의 Repository 직접 import | DB 접근 책임 경계 붕괴 방지 |
| 내부 인터페이스를 HTTP API처럼 설계 | 모듈러 모놀리식 내부 호출과 외부 계약 혼동 방지 |
| 공통 패키지에 도메인 규칙 배치 | 사실상 숨은 공유 도메인이 되는 문제 방지 |

### 5.3 패키지 공개 범위

각 도메인은 외부에 공개할 Port와 DTO를 명확히 분리한다.

```text
com.qtai.<domain>/
    api/              # 다른 도메인이 사용할 수 있는 Port, DTO
    application/      # UseCase, Command/Query handler
    domain/           # Entity, Value Object, Domain Service
    infrastructure/   # Repository 구현, 외부 API adapter
```

다른 도메인이 import할 수 있는 영역은 원칙적으로 `api/` 하위의 Port와 DTO뿐이다. `domain/`, `application/`, `infrastructure/` 하위 타입은 다른 도메인에서 import하지 않는다.

### 6.1 저장소 원칙

| 데이터 | 저장 원칙 |
| --- | --- |
| 성경 본문 | GitHub 공개 JSON을 확인 후 절 단위로 MySQL에 적재 |
| QT 본문 범위 | 공개 시각 00:00 KST, 수집 배치 04:00 KST. 범위 정보만 저장 |

| B 테이블 | Public Domain 영어 원문 주석. AI 재해석 입력 |
| C 테이블 | B를 AI가 한국어로 번역·재해석한 사용자 노출 해설 |
| AI 실행 이력 | `ai_runs`에 모델, 프롬프트 버전, 상태, 실패 사유 저장 |
| 편집자 검증 | `editor_verifications`에 검증 결과와 요약 저장 |
| 시뮬레이터 | `simulator_clips`에 클립 상태와 재사용 가능한 참조 저장 |
| 묵상 기록 | 묵상 노트와 묵상 달력 조회를 위한 기록 저장 |
| 이벤트 재처리 근거 | 필요 시 DB 상태와 eventId/error log 기준으로 재처리 |

### 6.2 성경 JSON 소스 관리

성경 JSON 소스는 적재 전에 별도 표로 관리한다. 이 표는 추후 ERD 또는 운영 문서와 연결할 수 있다.

| 항목 | 설명 |
| --- | --- |
| repo URL | GitHub 공개 JSON 저장소 URL |
| license | 데이터 라이선스 |
| translationName | 번역본명 |
| attribution | 출처 표기 방식 |
| redistributionAllowed | 재배포 가능 여부 |
| checkedAt | 검토 일시 |
| checkedBy | 검토자 |

개역개정, ESV, NIV는 사용하지 않는다. 검토 결과가 불명확한 데이터는 적재하지 않는다.

## 7. AI와 배치 구조

### 7.1 실행 원칙

| 원칙 | 내용 |
| --- | --- |
| 사용자 요청 경로 LLM 금지 | 사용자가 API를 호출할 때 LLM을 직접 호출하지 않는다. |
| 배치 우선 | 매일 04:00 KST QT 본문 범위 수집 이후 AI 배치를 실행한다. |
| 관리자 트리거 허용 | 관리자는 특정 본문의 해설·시뮬레이터 재생성을 수동 트리거할 수 있다. |
| 산출물 재사용 | 이미 성공한 `qtId + runType + promptVersion + inputHash` 조합은 재사용한다. |
| 산출물 버전 관리 | 관리자 재생성은 기존 산출물을 덮어쓰지 않고 새 version으로 저장한다. |

### 7.2 배치 흐름

```text
04:00 KST QT range collection
        |
        v
Bible text lookup from MySQL
        |
        v
B table source lookup
        |
        v
DeepSeek explanation generation
        |
        v
Editor verification
        |
        +-- PASS ----------> C table save
        +-- REJECT/FAILED -> ai_runs / verification log
        |
        v
Simulator generation or status update
        |
        v
Today QT cache update
        |
        v
AiContentGeneratedEvent publish
```

### 7.3 00:00~04:00 처리

성서 유니온의 QT 범위 공개 시각은 00:00 KST이고, 우리 시스템의 수집 배치 시각은 04:00 KST다.

00:00 KST부터 04:00 KST 전까지는 새 범위 수집과 AI 산출물이 완료되지 않았을 수 있으므로, 사용자는 이전에 준비된 캐시를 조회한다. 04:00 KST 배치 이후 성공한 산출물이 있으면 오늘 QT 캐시를 교체한다.

### 7.4 시뮬레이터 상태

오늘 QT 응답은 시뮬레이터 상태를 항상 반환한다. 실제 클립이 없거나 미완성인 경우에도 상태를 반환하고, 버튼 활성 여부는 상태로 결정한다.

| 상태 | 의미 | 사용자 버튼 |
| --- | --- | --- |
| `READY` | 클립 조회 가능 | 활성화 |
| `MISSING` | 아직 클립 없음 | 비활성화 |
| `FAILED` | 생성 실패 | 비활성화 |
| `DISABLED` | 해당 본문에서 제공하지 않음 | 비활성화 |

## 8. 인프로세스 이벤트 구조

v1은 Kafka를 사용하지 않고 Spring `ApplicationEventPublisher`를 사용한다. 이벤트는 모듈 간 상태 변경 알림, 캐시 무효화, 관리자 모니터링, 알림 큐잉을 위한 내부 통신 수단이다.

| 이벤트 | 발행 도메인 | 구독 도메인 | 목적 |
| --- | --- | --- | --- |
| `UserActivityTrackedEvent` | bff | bible | 구절 조회 이력 기록 |
| `AiContentGeneratedEvent` | ai | bible, bff, 관리자 | AI 해설(C)·시뮬레이터 산출물 적재 알림 |
| `EditorRejectedEvent` | ai | 관리자 | 편집자 에이전트 반려 알림 |
| `JournalCreatedEvent` | journal (bible 영역) | bff, journal | 노트 생성 완료, 묵상 달력 캐시 무효화 |
| `JournalUpdatedEvent` | journal (bible 영역) | bff, journal | 노트 수정 완료, 묵상 달력 캐시 무효화 |
| `JournalDeletedEvent` | journal (bible 영역) | bff, journal | 노트 삭제 완료, 묵상 달력 캐시 무효화 |
| `NotificationRequestedEvent` | 다수 | bff | 인앱 알림 큐잉 |

### 8.1 이벤트 실패 처리

Spring `ApplicationEventPublisher`는 영속 메시징을 보장하는 메시지 브로커가 아니다. v1은 핸들러 실패 로그와 재처리 가능성을 목표로 한다.

| 항목 | 기준 |
| --- | --- |
| 이벤트 식별 | `eventId`, `eventType`, `occurredAt`, 관련 aggregate id 기록 |
| 핸들러 실패 | error log와 실패한 handler 이름 기록 |
| 재처리 | DB 상태 기준으로 관리자 또는 배치가 재시도 |
| 묵상 이력 | 별도 journal event table에 영속 기록 |
| 외부 메시지 브로커 | v1에서는 도입하지 않음 |

| --- | --- | --- |
| 일반 사용자 | `USER` | 오늘 QT 조회, 묵상 기록, 찬양 목록 저장, 선택적 익명 공유 |
| 관리자 | `ADMIN` | 관리자 운영 API, 찬양 큐레이션, AI 산출물 재생성 트리거 |
| 시스템 작업 | `SYSTEM` | 배치, AI 생성/검증, 내부 재처리 작업 |

관리자 API는 `ADMIN` 권한을 요구한다. 배치와 AI 내부 작업은 사용자 권한으로 실행하지 않고 `SYSTEM` 작업으로 기록한다. 사용자 요청 경로에서는 LLM을 호출하지 않는다.

## 10. 배포와 운영 구조

| 항목 | v1 기준 |
| --- | --- |

| 로그 | 단일 로그 출력 후 Docker logs 수집 |

| 품질 게이트 | 도메인 import 검증, 테스트, 커버리지, 금지 기술 검사 |

Kubernetes, Helm, Kafka, 독립 마이크로서비스 운영은 v1 범위가 아니다. v2에서 분리할 수 있도록 도메인 경계와 Port/DTO 구조만 v1부터 강제한다.

## 11. 품질 게이트 연계

아키텍처 규칙은 문서 선언으로 끝내지 않고 `18_코드_품질_게이트.md`의 자동 검증과 연결한다.

| 규칙 | 검증 방향 |

| RAG/vector/ChromaDB/Elasticsearch 금지 | 의존성·설정·코드 키워드 검사 |
| Kafka/Kubernetes/Helm v1 도입 금지 | 의존성·워크플로·배포 파일 검사 |

| A 테이블 사용자 노출 금지 | API 응답 DTO와 Controller 검사 |
| 관리자 API 권한 | `ADMIN` 권한 테스트 |
| 배치/AI 작업 권한 | `SYSTEM` 기록과 사용자 요청 경로 LLM 미호출 테스트 |

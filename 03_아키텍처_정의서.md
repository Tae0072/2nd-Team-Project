## 3. 서비스별 책임·경계 (Bounded Context)

> Domain-Driven Design의 Bounded Context를 6개로 분리. 각 Context는 **자기 어휘(Ubiquitous Language)**를 가진다.

### 3.1 Auth/User Context (Owner: 강상민)

| 항목 | 내용 |
| --- | --- |
| **핵심 어휘** | User, Email, PasswordHash, RefreshToken, OAuthLink, JWT, Role |
| **외부에 노출하는 것** | `users.id` (BIGINT), `email`은 마스킹 후 외부 노출 가능 |
| **외부에 노출하지 않는 것** | `password_hash`, `token_hash`, JWT 개인키 |
| **다른 Context와 연결** | Kafka `user.deactivated` 발행 → AI·Journal 정리 |
| **API 책임** | POST /auth/register, POST /auth/login, POST /auth/refresh, POST /auth/logout, DELETE /auth/me, POST /auth/oauth/google |
| **Redis-WS 사용** | refresh token blacklist (`auth:refresh:revoked:{jti}`, TTL = refresh 만료까지). Access는 만료까지 그냥 유효 (blacklist 없음) |
| **금지 사항** | 다른 Context의 user_id를 자체 검증 위해 직접 조회 금지 (Gateway가 JWT로 신뢰) |

### 3.2 Bible Context (Owner: 김태혁)

| 항목 | 내용 |
| --- | --- |
| **핵심 어휘** | Book, Chapter, Verse, Version (KJV/REVISED), Commentary, Source, Language, Passage |
| **외부에 노출** | `books.id`, passage 좌표 (book_code, chapter, verse) |
| **외부에 노출하지 않음** | 내부 ETL 메타·캐시 키 |

## 4. 통신 패턴 — Sync vs Async 결정 매트릭스

### 4.1 결정 매트릭스

| 상황 | 패턴 | 이유 | 예시 |
| --- | --- | --- | --- |
| 사용자가 즉시 응답을 기다림 | **Sync REST** | 응답 시간 = 사용자 인지 시간 | Flutter → Gateway → BFF → Bible (passage 조회) |
| LLM 응답 같은 긴 응답 | **Sync SSE** | 첫 토큰부터 점진 노출 (P95 ≤ 2000ms 목표) | Flutter → Gateway → AI Service (스트리밍) |
| 사용자에게 즉시 푸시 | **Sync WebSocket** | 서버 → 클라이언트 push | BFF → Flutter (notification) |
| 한 도메인의 변경이 다른 도메인을 트리거 | **Async Kafka 이벤트** | 결합도 ↓, 장애 격리, 재처리 가능 | AI 세션 완료 → Journal 자동 생성 |
| 통계·집계·로그 적재 | **Async Kafka 이벤트** | 본 흐름과 분리, 부하 격리 | 사용자 활동 → Journal·통계 적재 |
| 보상 트랜잭션 (실패 처리) | **Async Kafka 이벤트** | Saga 패턴 | journal.creation.failed 보상 |

### 4.2 동기 호출 표준 (Sync)

| 항목 | 표준 |
| --- | --- |
| 프로토콜 | HTTP/1.1 (v1.0) — gRPC는 v2.0 검토 |
| 클라이언트 | **Spring 6 RestClient** (또는 WebClient for SSE) |
| **Service Discovery** | **K8s DNS** — base URL 패턴: `http://{service-name}.{namespace}.svc.cluster.local:8080` (예: `http://auth-service.qtai.svc.cluster.local:8080`). 같은 namespace면 `http://auth-service:8080` 단축 형도 OK |
| 타임아웃 | connect 1s / read 800ms (서비스 간) / read 5s (LLM 첫 토큰) — **참고:** Anthropic Claude 평균 첫 토큰 1-2s, P95 더 김 → SLO 2000ms는 **stretch goal** (§ 15.1) |
| 재시도 | 1회 (5xx에 한해), exponential backoff 100ms |
| Circuit Breaker | **Resilience4j** — 50% 실패율 10s 윈도우 → OPEN 30s |
| 인증 | Gateway에서 JWT 검증 후 내부는 신뢰 + **NetworkPolicy로 namespace 격리** (mTLS는 v2.0 검토) |
| traceId | 모든 호출에 W3C Trace Context 헤더 (`traceparent`) 전파 |

### 4.3 비동기 이벤트 표준 (Async)

| 항목 | 표준 |
| --- | --- |
| 메시지 브로커 | **Apache Kafka KRaft 모드** (v1.0 single-node = 1 broker, v1.1 multi-broker 검토) |
| 스키마 검증 | **Apicurio Schema Registry v2.5+** (JSON Schema, eventType + eventVersion) |
| Producer 설정 | `acks=all`, `enable.idempotence=true`, `compression.type=snappy`. **v1.0 single-node에서 acks=all == acks=1** (의미 약함, 학습·검증용) |
| Consumer 설정 | `enable.auto.commit=false`, manual ack, `max.poll.records=10` |
| **Consumer Group 명명** | `{service}-{topic-purpose}-consumer` (예: `journal-service-ai-completion-consumer`) |
| 멱등성 | DB UNIQUE 제약 (`idempotency_key`)으로 강제 |
| 재시도 | Spring Kafka `RetryableTopic` 3회 → DLQ |
| DLQ | `{topic}.DLQ` (자동 생성) |
| **이벤트 발행 v1.0** | **`@TransactionalEventListener(phase = AFTER_COMMIT)` + KafkaTemplate** (TX 커밋 후 발행, publish 실패 시 메트릭 + 24h 보상 배치 — § 6.3) |

### 4.4 SSE / WebSocket 표준

| 패턴 | 사용처 | 표준 |
| --- | --- | --- |
| **SSE** (Server-Sent Events) | AI 응답 스트리밍 (서버 → 클라이언트 단방향) | Spring `Flux<ServerSentEvent>` + Gateway 패스스루. 첫 토큰 timeout 5s, 토큰 간 idle timeout 30s, 전체 max 5분. **`Flux.doFinally(CANCEL)`로 클라이언트 끊김 시 LLM 호출 중단** |
| **WebSocket** | 실시간 알림 (양방향, 다중 메시지) | Spring `@MessageMapping` (STOMP) + Redis-WS pub/sub로 BFF Pod 간 세션 공유. 30s ping |

### 4.5 결정 플로우 차트

```
[새로운 통신 요구]
       │
       ▼
사용자가 즉시 결과를 봐야 함?
   │
   ├─ YES ──▶ 응답 본문이 큼 (LLM 등)?
   │              │
   │              ├─ YES ──▶ SSE

                  └─ NO ───▶ 그냥 로그·메트릭에만 적재
```

```java
// usecase/GetPassageAggregateUseCase.java
public interface GetPassageAggregateUseCase {
    PassageAggregateResponse execute(String book, int chapter, int verse, Long userId);
}

@Service
@RequiredArgsConstructor
public class GetPassageAggregateUseCaseImpl implements GetPassageAggregateUseCase {

    private final BibleServiceClient bibleClient;
    private final ActivityEventPublisher eventPublisher;  // Kafka 발행 어댑터

    @Override
    public PassageAggregateResponse execute(String book, int chapter, int verse, Long userId) {
        // 병렬 호출 3개 (CompletableFuture) — KR + EN + Commentary
        CompletableFuture<KrPassage> krFuture = bibleClient.getKrAsync(book, chapter, verse);
        CompletableFuture<EnPassage> enFuture = bibleClient.getEnAsync(book, chapter, verse);
        CompletableFuture<List<Commentary>> commFuture = bibleClient.getCommentariesAsync(book, chapter, verse);

        CompletableFuture.allOf(krFuture, enFuture, commFuture).join();

        // 사용자 활동 이벤트 발행 (Kafka — Journal Service가 비동기 적재)
        // ⚠️ ApplicationEventPublisher가 아니라 KafkaTemplate 사용
        eventPublisher.publishUserActivity(
            userId, "READ_PASSAGE", book, chapter, verse, Instant.now()
        );

        return PassageAggregateResponse.of(krFuture.get(), enFuture.get(), commFuture.get());
    }
}

// kafka/ActivityEventPublisher.java
@Component
@RequiredArgsConstructor
public class ActivityEventPublisher {
    private final KafkaTemplate<String, UserActivityEvent> kafkaTemplate;

    public void publishUserActivity(Long userId, String type, String book, int ch, int v, Instant at) {
        UserActivityEvent event = UserActivityEvent.builder()
            .eventId("evt_" + UlidCreator.getUlid())
            .eventType("user.activity.tracked")
            .eventVersion(1)
            .occurredAt(at)
            .traceId(MDC.get("traceId"))
            .producerService("bff-aggregator")
            .data(new UserActivityData(userId, type, book, ch, v, at))
            .build();
        
        String key = String.valueOf(userId);  // 같은 user는 같은 partition
        kafkaTemplate.send("user.activity.tracked", key, event)
            .whenComplete((result, ex) -> {
                if (ex != null) {
                    log.error("Failed to publish user.activity.tracked: {}", ex.getMessage(), ex);
                    Metrics.counter("kafka_publish_failure_total", "topic", "user.activity.tracked").increment();
                }
            });
    }
}
```

### 5.4 Journal Service 이벤트 소싱 패턴 (이승욱) — v1.1 정정

> **v1.0 → v1.1 핵심 변경:** TX 안에서 직접 `kafkaTemplate.send()` 호출하지 않고, `@TransactionalEventListener(AFTER_COMMIT)` 패턴으로 변경. TX 커밋 성공 후에만 publish 시도 → 일관성 ↑.

```java
// domain/journal/JournalService.java
@Service
@RequiredArgsConstructor
public class JournalService {

    private final JournalRepository journalRepo;
    private final JournalEventRepository eventRepo;
    private final ApplicationEventPublisher springEventPublisher;  // Spring 내부 이벤트 (TX 콜백용)

    @Transactional
    public Long appendEvent(Long journalId, EventType type, JsonNode data, String idempotencyKey) {
        // 1. JOURNALS 행 잠금 — @Lock(PESSIMISTIC_WRITE) 사용
        //    (Repository에서 @Lock 명시 메서드를 따로 두는 게 표준)
        Journal journal = journalRepo.findByIdLocked(journalId)
            .orElseThrow(() -> new BusinessException(JOURNAL_NOT_FOUND));

        // 2. last_event_sequence 증가
        long nextSeq = journal.getLastEventSequence() + 1;
        journal.setLastEventSequence(nextSeq);

        // 3. JOURNAL_EVENTS INSERT (idempotency_key UNIQUE + (journal_id, sequence) UNIQUE)
        JournalEvent event = JournalEvent.builder()
            .journalId(journalId)
            .eventType(type)
            .eventData(data)
            .idempotencyKey(idempotencyKey)
            .sequence(nextSeq)
            .build();
        eventRepo.save(event);

        // 4. read model 갱신
        journal.applyEvent(event);

        // 5. Spring 내부 이벤트 발행 (실제 Kafka publish는 TX AFTER_COMMIT에서)
        springEventPublisher.publishEvent(new JournalEventAppendedEvent(event));

        return event.getId();
    }
}

// domain/journal/JournalRepository.java — @Lock 명시 메서드
public interface JournalRepository extends JpaRepository<Journal, Long> {
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT j FROM Journal j WHERE j.id = :id")
    Optional<Journal> findByIdLocked(@Param("id") Long id);
}

// infra/kafka/JournalEventKafkaPublisher.java — TX 커밋 후 발행
@Component
@RequiredArgsConstructor
public class JournalEventKafkaPublisher {
    private final KafkaTemplate<String, JournalEventMessage> kafkaTemplate;

    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void on(JournalEventAppendedEvent event) {
        JournalEventMessage msg = JournalEventMessage.from(event.journalEvent());
        String topic = "journal." + msg.eventType().toLowerCase();  // journal.created / journal.updated / ...
        String key = String.valueOf(msg.journalId());
        
        kafkaTemplate.send(topic, key, msg)
            .whenComplete((result, ex) -> {
                if (ex != null) {
                    log.error("Publish failed: topic={}, key={}, err={}", topic, key, ex.getMessage(), ex);
                    Metrics.counter("kafka_publish_failure_total", "topic", topic).increment();
                    // v1.0 한계: TX 이미 커밋됨 + publish 실패 → 24h 보상 배치 또는 v1.1 Outbox
                }
            });
    }
}
```

**왜 이 패턴인가:**
- TX 안에서 외부 시스템(Kafka) 호출 시 외부 실패 → TX 롤백? Kafka는 트랜잭션 매니저로 묶기 어려움
- `AFTER_COMMIT`: TX 커밋 성공 후에만 publish 시도 → DB 상태와 발행 메시지 일치 보장
- 단점: 발행 자체 실패 시 메시지 유실 (v1.0 한계, § 11.2.1) → v1.1 Outbox로 본질 해결

## 6. Saga 패턴 + Outbox 전략

### 6.1 적용 시나리오

> AI 대화 완료 → Journal 자동 생성 — 둘 중 하나 실패 시 보상 필요.

```
정상 흐름:
[1] AI Service: AI_SESSIONS.status=COMPLETED + summary 생성 (로컬 TX)
[2] AI Service: AFTER_COMMIT → ai.session.completed 이벤트 발행
[3] Journal Service: 컨슈머 수신 → JOURNALS DRAFT 생성 + JOURNAL_EVENTS CREATED 적재 (멱등성 키 검증)

실패 흐름 1 — Journal 1차 처리 실패:
[3a] Spring Kafka RetryableTopic 자동 재시도 3회 (backoff 1s, 5s, 25s)
[3b] 영구 실패 시 ai.session.completed.DLQ로 이동
[3c] @DltHandler가 journal.creation.failed 발행 (Saga 보상)
[3d] AI Service 컨슈머: AI_SESSIONS.status='COMPLETED_NO_JOURNAL' 마킹

실패 흐름 2 — AI Service publish 실패 (v1.0 한계):
[2a] AFTER_COMMIT 후 publish 실패 → ERROR 로그 + Prometheus(kafka_publish_failure_total)
[2b] 24h 배치: AI_SESSIONS 중 status=COMPLETED인데 Journal 없는 행 재발행
v1.1 Outbox 패턴으로 본질 해결
```

### 6.2 Saga 구현 표준 (Choreography-based, v1.0)

**원칙:**
- Orchestrator 별도 두지 않음 — 각 Service가 이벤트로 자율 협력
- 각 단계는 **로컬 TX** (서비스 내) + **AFTER_COMMIT 이벤트 발행** (서비스 간)
- 보상은 **반대 방향 이벤트** (`xxx.failed`, `xxx.compensated`)

**책임 분배:**

| Service | Saga에서 역할 | 보상 행동 |
| --- | --- | --- |
| AI Service | Saga 시작 (session.completed) | status=COMPLETED_NO_JOURNAL 마킹 |
| Journal Service | Saga 컨슈머 (journal 생성) | DLQ 적재 + journal.creation.failed 발행 |

### 6.3 Outbox 패턴 — v1.0 한계 + v1.1 본질 해결

**v1.0 한계 (이미 02번 § 11.2.1에 박제):**
- AFTER_COMMIT 후 publish 실패 시 이벤트 유실
- 대응: ① 발행 실패 메트릭 알림 ② 24h 보상 배치 ③ W4 부하 테스트 검증

**v1.1 본질 해결 (Outbox 패턴):**

```
[AI Service]
  TX 시작
    ├─ AI_SESSIONS UPDATE
    └─ OUTBOX INSERT (이벤트 페이로드)
  TX 커밋  ← DB 상태와 OUTBOX가 한 트랜잭션에 묶임 (atomicity 보장)
       │
       ▼
  [Outbox 폴러] (v1.1: AI Service Pod 안의 별도 스레드 — 같은 Pod이라 K8s 자원 절약)
       │ 1초마다 폴링
       ▼
  Kafka publish
       │ 성공 시 OUTBOX status=PUBLISHED
       │ 실패 시 status=FAILED, retry_count++ 후 재시도
```

**Outbox 테이블 스키마 (v1.1 추가 예정):**
```sql
CREATE TABLE outbox (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    aggregate_type VARCHAR(50),
    aggregate_id BIGINT,
    event_type VARCHAR(100),
    topic VARCHAR(100),
    payload JSON,
    status VARCHAR(20) DEFAULT 'PENDING',  -- PENDING/PUBLISHED/FAILED
    created_at DATETIME(6),
    published_at DATETIME(6),
    retry_count INT DEFAULT 0,
    INDEX idx_outbox_status_created (status, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

**v2.0 검토:** Debezium CDC로 OUTBOX 폴링 대체.

### 7.3 호환성 정책

| 정책 | 설정 | 의미 |
| --- | --- | --- |
| **BACKWARD** (default) | producer 신 버전 + consumer 구 버전 OK | 새 필드는 optional만 추가 |
| FORWARD | producer 구 버전 + consumer 신 버전 OK | 필드 제거만 가능 |
| FULL | 양방향 | 가장 안전, 가장 제약 큼 |
| NONE | 검증 없음 | v1.0 초기 prototyping용으로만 |

v1.0 표준: **BACKWARD** — 컨슈머가 천천히 따라잡아도 OK.

### 7.5 Consumer 표준 설정 + RetryableTopic (v1.1 클래스 정정)

```java
// 사용자 정의 예외 (도메인별로 분리)
public class TransientProcessingException extends RuntimeException { ... }
public class PermanentValidationException extends RuntimeException { ... }

@RetryableTopic(
    attempts = "3",
    backoff = @Backoff(delay = 1000, multiplier = 5.0),  // 1s, 5s, 25s
    autoCreateTopics = "true",
    dltStrategy = DltStrategy.FAIL_ON_ERROR,
    // 일시 오류만 재시도 — 사용자 정의 또는 Spring Data 표준
    include = {
        TransientProcessingException.class,
        org.springframework.dao.TransientDataAccessException.class
    },
    // 검증 오류는 즉시 DLQ (재시도해도 결과 동일)
    exclude = {
        PermanentValidationException.class,
        com.fasterxml.jackson.core.JsonProcessingException.class
    }
)
@KafkaListener(
    topics = "ai.session.completed",
    groupId = "journal-service-ai-completion-consumer"
)
public void handle(SessionCompletedEvent event) {
    String idempotencyKey = "ai.session.completed:" + event.getSessionId();
    try {
        journalService.createFromAiSession(event, idempotencyKey);
    } catch (DataIntegrityViolationException e) {
        // UNIQUE 위반 = 멱등성 차단 작동 = 정상 무시
        log.info("Duplicate event skipped: {}", idempotencyKey);
    }
}

@DltHandler  // DLQ 도착 시 호출
public void handleDlt(SessionCompletedEvent event,
                      @Header(KafkaHeaders.EXCEPTION_MESSAGE) String error) {
    log.error("DLT received: {}, error: {}", event, error);
    // journal.creation.failed 이벤트 발행 (Saga 보상 시작)
    publishCompensation(event);
}
```

### 7.6 DLQ 모니터링

- 모든 DLQ 토픽은 **Prometheus consumer lag 메트릭**으로 모니터링 (`kafka_consumer_lag`)
- 메시지 1건이라도 들어오면 알림 (Grafana Alert)
- 매주 금 통합 미팅에서 DLQ 잔여 메시지 수 공유 (강태오)

### 9.3 ConfigMap vs Secret 결정 규칙

| 데이터 종류 | 어디에 | 예 |
| --- | --- | --- |
| 비-민감 설정 (URL, 타임아웃, feature flag) | **ConfigMap** | `KAFKA_BOOTSTRAP_SERVERS`, `LOG_LEVEL` |

**v1.0 시크릿 관리:**
- K8s Secret 직접 생성 (`kubectl create secret`)

- Helm values에 평문 시크릿 X — values 파일에는 `valueFrom: secretKeyRef`만

## 11. 보안 · 시크릿 관리

### 11.1 인증 방식 — JWT (RS256)

| 항목 | 표준 |
| --- | --- |
| 알고리즘 | **RS256** (비대칭) — Gateway는 공개키만, Auth Service만 개인키 |
| Access Token 만료 | 30분 |
| Refresh Token 만료 | 14일 (v1.1에 7일 검토 — 보안 강화 환경) |
| Refresh 방식 | Token Rotation (사용 시 새 토큰 발급 + 이전 revoke) |
| Token 저장 (클라이언트) | Flutter `flutter_secure_storage` (KeyChain/Keystore) |
| Token 저장 (서버) | Refresh만 hash로 저장 (`token_hash`), Access는 저장 X |
| 폐기 | logout 시 **Refresh의 jti만 Redis blacklist** (`auth:refresh:revoked:{jti}`, TTL = refresh 만료까지). **Access는 blacklist 없음** — 만료(30분)까지 유효 (단명이라 위험 낮음) |

**JWT Claim 구조:**
```json
{
  "iss": "qtai-auth",
  "sub": "5678",                        // user_id
  "iat": 1716700800,
  "exp": 1716702600,                    // iat + 1800s (Access)
  "jti": "tok_01HZX...",                // unique ID (Refresh blacklist 키)
  "scope": "ROLE_USER",
  "tier": "free"                        // 추후 프리미엄 구분
}
```

**JwtAuthenticationConverter 설정 — scope claim → ROLE_* GrantedAuthority 매핑:**

```java
@Configuration
@EnableWebSecurity
@EnableMethodSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .oauth2ResourceServer(oauth2 -> oauth2
                .jwt(jwt -> jwt.jwtAuthenticationConverter(jwtAuthenticationConverter()))
            )
            // ... 다른 설정
        ;
        return http.build();
    }

    @Bean
    public JwtAuthenticationConverter jwtAuthenticationConverter() {
        JwtGrantedAuthoritiesConverter authConverter = new JwtGrantedAuthoritiesConverter();
        authConverter.setAuthorityPrefix("");           // 우리 claim에 이미 ROLE_ prefix 있음
        authConverter.setAuthoritiesClaimName("scope"); // claim 이름

        JwtAuthenticationConverter conv = new JwtAuthenticationConverter();
        conv.setJwtGrantedAuthoritiesConverter(authConverter);
        conv.setPrincipalClaimName("sub");
        return conv;
    }
}
```

### 11.3 인증 시나리오

```
[로그인]
Flutter ─POST /auth/login {email, password}─▶ Gateway ─▶ Auth Service
                                                              │
                                                              ├─ bcrypt 검증
                                                              ├─ Access Token (RS256, 30분) 생성
                                                              ├─ Refresh Token 생성 + token_hash DB 저장
                                                              └─ {accessToken, refreshToken} 응답
                                                              
[보호된 리소스 접근]
Flutter ─Authorization: Bearer {accessToken}─▶ Gateway
                                                  │
                                                  ├─ JWT 검증 (RS256 공개키)
                                                  ├─ exp 만료 확인
                                                  └─ 통과 시 sub(user_id)를 X-User-Id 헤더로 내부 라우팅
                                                     (NetworkPolicy로 외부 spoofing 차단)

[Refresh]
Flutter (Access 만료 감지) ─POST /auth/refresh {refreshToken}─▶ Gateway ─▶ Auth Service
                                                                              │
                                                                              ├─ token_hash DB 조회
                                                                              ├─ Redis-WS auth:refresh:revoked:{jti} 검사
                                                                              ├─ 만료·revoke 확인
                                                                              ├─ Token Rotation: 기존 Redis blacklist 추가 + 새 토큰 발급
                                                                              └─ {accessToken, refreshToken} 응답
```

### 11.4 권한 모델

| Role | 권한 |
| --- | --- |
| 비로그인 (anonymous) | 공개 페이지만 (랜딩) |
| ROLE_USER | 본인 데이터 CRUD, AI 대화, Journal |
| ROLE_ADMIN | 위 + 대시보드, 통계, 사용자 제재 |

**Method-level Security (v1.1 정정 — `hasAuthority` 사용):**

```java
// hasAuthority('ROLE_ADMIN')이 명확
@PreAuthorize("hasAuthority('ROLE_ADMIN')")
@GetMapping("/admin/dashboard")
public DashboardResponse getDashboard() { ... }

@PreAuthorize("@journalSecurity.isOwner(#id, authentication)")
```

### 14.3 핵심 ADR 본문 — 0001 (서비스 경계)

> 본 § 14.3·§ 14.4의 본문은 **요약본**. 정식 ADR 본문은 `docs/adr/0001-*.md`·`docs/adr/0007-*.md`에 1대1로 동일 내용 복제 (§ 14.5).

```markdown
# ADR-0001: MSA 서비스 경계 (6 services)

## 상태
Accepted (사후 정리)

## Context
1차(HMS) 모놀리식 대비 2차(QT-AI)는 MSA 풀스코프를 학습 목표로 한다. Bounded Context를 잘못 자르면 W2 이후 재설계 비용이 폭증하므로 W1 안에 결정해야 한다.

## Decision
다음 6개 서비스로 분리한다:
- API Gateway (인프라)
- BFF Aggregator (인프라)
- Auth/User Service (도메인)
- Bible Service (도메인)
- AI/RAG Service (도메인)
- Journal Service (도메인)

## Alternatives
- 4 services (Auth + Bible + AI/Journal 통합 + Gateway): Journal과 AI의 책임이 섞여 이벤트 소싱 적용이 어려움
- 8 services (각 도메인을 더 잘게): 6명에 비해 운영 복잡도 ↑, W1 안 끝낼 가능성 ↑
- 모놀리식: 1차와 동일, 학습 목표 미달

## Consequences
- (+) 각 owner가 1 service end-to-end 책임 — Service Owner 모델 가능
- (+) 도메인 격리 — 이벤트 소싱·Saga·캐시 전략 분리 적용 가능
- (-) Cross-cutting 변경(예: BaseEntity) 시 6 service 동시 sync 필요 (ADR-0009 대응)
- (-) 분산 디버깅 비용 ↑ (Foundation Lock-in 5 관측성으로 대응)

## 검증 방법
- W1 5/22 금: 6 service 모두 hello-world 배포 + 서비스 간 ping 성공
- W2 5/26~29: 각 owner가 자기 service 도메인 코드 충돌 없이 병렬 머지
```

### 14.4 핵심 ADR 본문 — 0007 (Kafka 멱등성)

```markdown
# ADR-0007: Kafka 멱등성 키 DB UNIQUE 제약 강제

## 상태
Accepted (사후 정리, 02번 v1.1에서 결정)

## Context
컨슈머 코드 신뢰만으로는 중복 처리 위험이 크다. (특히 이승욱 작업의 under-deliver 경향상 코드 레벨 검사가 누락될 수 있다 — 운영 노트 § 3 참조). DB 제약으로 강제하면 사람의 실수가 차단된다.

## Decision
모든 외부 이벤트 컨슈머는 자기 측 DB에 `idempotency_key VARCHAR UNIQUE` 컬럼을 두고, 처리 시 INSERT 시도 → DUPLICATE KEY 발생 시 정상 무시한다.

대표 예: `JOURNAL_EVENTS.idempotency_key UNIQUE`.

## Alternatives
- 컨슈머 코드 레벨 검사 (Redis 캐시·메모리 Set): 코드 누락 시 무방어, 캐시 만료 시 중복
- Kafka exactly-once semantics (transactional producer + consumer): v1.0에 과도, 학습 곡선 ↑

## Consequences
- (+) 사람 실수와 무관하게 멱등성 보장
- (+) DLQ 재처리 시에도 안전
- (-) DB 부하 약간 ↑ (UNIQUE INDEX)
- (-) idempotency_key 표준화 필요 (§ 7.1 토픽 표에 박제)

## 검증 방법
- 통합 테스트: 같은 이벤트 2회 발행 → 컨슈머가 1번만 처리 (DB row 1개 + 1번은 idempotent skip 로그)
- W4 부하 테스트: 동일 이벤트 100회 동시 발행 → DB row 1개 + 99회 skip (assert)
```

## 3. 인증 (Authentication)

### 3.1 인증 방식 (v1.0)

| 방식 | 적용 | 토큰 |
| --- | --- | --- |
| **ID/PW 로그인** | 회원가입 사용자 | JWT (Access 30min + Refresh 14d) |
| **Google OAuth (ID Token)** | 소셜 로그인 사용자 | JWT (위와 동일) |
| **MFA** | ❌ v1.0 미지원 | v1.1 ADR-0013 후보 |
| **Passwordless (Magic Link)** | ❌ v1.0 미지원 | v2.0 검토 |

### 3.2 비밀번호 정책

| 항목 | 정책 |
| --- | --- |
| 최소 길이 | 8자 |
| 최대 길이 | 72자 (bcrypt 한계) |
| 복잡도 | 영문 + 숫자 + 특수문자 1개 이상 (위반 시 04번 § 4.2 `PASSWORD_TOO_WEAK`) |
| 해싱 | **bcrypt cost=12** (~250ms/hash on Pod 500m CPU) |
| 솔트 | bcrypt 자동 (16바이트) |
| 평문 저장 | **절대 금지** (DB 컬럼은 `password_hash VARCHAR(100)` — 02번 § 2.2) |
| 비교 | `BCryptPasswordEncoder.matches()` 만 사용 (timing attack 방어) |
| 만료·강제 변경 | ❌ v1.0 미지원 (UX 우선). v1.1 검토 |

```java
// Gateway Auth AuthService.java
@Service
public class AuthService {
    private final BCryptPasswordEncoder encoder = new BCryptPasswordEncoder(12);

    public User register(String email, String password, String nickname) {
        validatePasswordPolicy(password);  // 길이·복잡도 검증
        String hash = encoder.encode(password);
        return userRepository.save(new User(email, nickname, hash));
    }

    public boolean login(String email, String rawPassword) {
        User user = userRepository.findActiveByEmail(email)
            .orElseThrow(() -> new InvalidCredentialsException("INVALID_CREDENTIALS"));
        // 항상 encoder.matches() 호출 — 사용자 존재 여부로 응답 시간 다름 → timing 공격 방어
        return encoder.matches(rawPassword, user.getPasswordHash());
    }
}
```

### 3.3 JWT 표준

#### 3.3.1 알고리즘 + 키

| 항목 | 표준 |
| --- | --- |
| 알고리즘 | **RS256** (RSA-2048 비대칭 서명) |
| Private Key | Gateway Auth만 보유 (서명용) |
| Public Key | 모든 service + Gateway에 배포 (검증용) |
| 키 형식 | PEM (PKCS#8 private, X.509 public) |
| 키 길이 | 2048 bit (v1.0 충분, v1.1에 4096 검토) |

> **HS256 사용 안 함:** 대칭 키는 Gateway Auth와 Gateway·각 service 모두가 같은 secret을 가져야 함 → secret 다중 노출 위험. RS256은 Auth만 private 보관.

#### 3.3.2 Access Token 페이로드

```json
{
  "iss": "qtai-auth",
  "sub": "5678",
  "aud": "qtai-api",
  "iat": 1715140800,
  "exp": 1715142600,
  "jti": "01HZX...",
  "role": "ROLE_USER",
  "email_verified": true
}
```

| Claim | 의미 | 검증 |
| --- | --- | --- |
| `iss` | issuer | Gateway가 `qtai-auth` 일치 검증 |
| `sub` | subject (user_id) | Gateway가 X-User-Id 헤더에 주입 |
| `aud` | audience | Gateway가 `qtai-api` 일치 검증 |
| `iat` | issued at | 정보용 |
| `exp` | expiration | Gateway가 만료 검증 (시계 skew 30s 허용) |
| `jti` | JWT ID (ULID) | blacklist 조회 키 |
| `role` | RBAC role | service 측 `@PreAuthorize` 사용 |
| `email_verified` | 이메일 인증 여부 | (v1.1 이메일 인증 도입 시) |

> **민감 정보 절대 금지:** 비밀번호·이메일 본문·실명 등은 JWT에 넣지 않음. JWT는 base64 디코드만으로 누구나 읽음.

#### 3.3.3 Refresh Token

| 항목 | 표준 |
| --- | --- |
| 만료 | 14일 (v1.1에 7일 검토) |
| 저장 | DB `REFRESH_TOKENS.token_hash` (SHA-256 해시 저장 — 원본 X) |
| Rotation | 매 refresh 호출 시 새 발급 + 기존 revoke (04번 § 4.4) |
| Race Condition 방어 | Redis Distributed Lock `auth:refresh:rotating:{jti}` (NX, TTL 5s — 04번 § 4.4) |
| Blacklist | Redis `auth:refresh:revoked:{jti}` SET (TTL = 만료까지) |
| Logout | DB `revoked_at` UPDATE + Redis blacklist 등록 (04번 § 4.5 인증 ❌) |

### 3.4 JWT 키 운영 (Critical)

#### 3.4.1 키 생성 (W0)

```bash
# RSA-2048 private/public key pair 생성
openssl genrsa -out jwt-private.pem 2048
openssl rsa -in jwt-private.pem -pubout -out jwt-public.pem

# K8s Secret으로 적재 (qtai namespace)
kubectl create secret generic jwt-keys \
  --from-file=private.pem=jwt-private.pem \
  --from-file=public.pem=jwt-public.pem \
  --namespace=qtai

# 평문 키 파일은 즉시 삭제
shred -u jwt-private.pem jwt-public.pem    # Linux
# 또는 sdelete jwt-private.pem jwt-public.pem    # Windows
```

> **로컬 디스크에 .pem 파일 절대 보관 금지.** 생성 직후 K8s Secret 적재 → 로컬 파일 영구 삭제. `.gitignore`에 `*.pem` 등록 (이미 있음 — § 5.2).

#### 3.4.2 키 마운트 (Pod 측)

```yaml
# helm/qtai-umbrella/templates/auth-deployment.yaml (개념)
spec:
  template:
    spec:
      containers:
      - name: gateway
        env:
        - name: JWT_PRIVATE_KEY_PATH
          value: /etc/qtai/jwt/private.pem
        volumeMounts:
        - name: jwt-keys
          mountPath: /etc/qtai/jwt
          readOnly: true
      volumes:
      - name: jwt-keys
        secret:
          secretName: jwt-keys
          defaultMode: 0400              # 소유자 읽기만
```

> **Gateway Auth만 private.pem 마운트.** Gateway·다른 service는 `public.pem`만 마운트 — least privilege.

#### 3.4.3 키 회전 절차 (v1.0 — 수동)

> v1.0은 자동 회전 미구현. 다음 절차로 분기별 1회 또는 사고 시 회전.

1. 새 RSA 키 쌍 생성 (`jwt-private-v2.pem`, `jwt-public-v2.pem`)
2. K8s Secret에 v2 추가 (기존 v1 유지)
3. Gateway Auth kid (Key ID) 헤더에 `v2` 발급 시작 — JWT 페이로드 `kid` claim 추가
4. Gateway·service는 v1 + v2 둘 다 검증 가능 (JWKS endpoint 또는 K8s Secret로 둘 다 마운트)
5. 14일 + α 경과 후 (모든 v1 refresh token 만료) v1 키 K8s Secret에서 제거
6. 절차 모두 강태오 + 이지윤 페어 작업 (single point 회전 사고 방어)

> **v1.1 자동 회전:** Bitnami Sealed Secrets 또는 ExternalSecrets + AWS Secrets Manager 도입 검토. ADR-0014 후보.

### 3.5 OAuth (Google) — JWK 직접 검증 (04번 § 4.8)

#### 3.5.1 흐름

```
[Flutter]   1. Google Sign-In SDK → ID Token 획득
            2. POST /api/v1/auth/oauth/google { idToken }
                   ↓
[Auth]      3. Google JWKS endpoint 캐시 조회
                  GET https://www.googleapis.com/oauth2/v3/certs (1h cache)
            4. ID Token JWT 직접 검증 (nimbus-jose-jwt):
                 - 서명 (RS256, JWKS의 kid 매치)
                 - iss: "accounts.google.com" 또는 "https://accounts.google.com"
                 - aud: 우리 Google Client ID (K8s Secret 로드)
                 - exp: 만료 검증
                 - email_verified: true 확인
            5. claim에서 sub (Google user_id), email 추출
            6. OAUTH_LINKS (provider='GOOGLE', provider_user_id=sub) 조회
                 - 매치 → 로그인 (USERS 조회)
                 - 없음 → 새 USERS + OAUTH_LINKS 생성 (이메일 마스킹 정책 적용 — 02번 § 2.2.1)
            7. 우리 JWT (Access + Refresh) 발급 → 응답
```

#### 3.5.2 보안 검증 포인트

| 검증 | 위반 시 |
| --- | --- |
| 서명 (kid 매치 + Google 공개키) | 401 `OAUTH_TOKEN_INVALID` |
| iss = `accounts.google.com` 또는 `https://accounts.google.com` | 401 `OAUTH_TOKEN_INVALID` |
| aud = 우리 Google Client ID | 401 `OAUTH_TOKEN_INVALID` (다른 앱 토큰 spoofing 차단) |
| exp 미만료 | 401 `OAUTH_TOKEN_INVALID` |
| email_verified = true | 401 `OAUTH_TOKEN_INVALID` (미인증 이메일은 도메인 spoofing 가능) |
| nonce 검증 | v1.0 미적용 (Flutter SDK 측 nonce 생성 검토 v1.1) |

> **`tokeninfo` endpoint 사용 금지 (v1.1):** Google이 production 환경에서 비권장. 추가 round-trip + provider 의존성. JWK 직접 검증이 표준.

> **Google Client Secret:** v1.0은 server-side flow 안 씀 (Flutter가 ID Token만 보냄). Client Secret 시크릿 적재는 Flutter 측 검토 — 일반적으로 client_id만 사용하고 client_secret은 서버 flow 전용.

### 3.6 Gateway JWT 검증 흐름

```java
// Gateway JwtAuthenticationFilter (개념 — Spring Cloud Gateway WebFlux)
@Component
public class JwtAuthFilter implements GlobalFilter {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest req = exchange.getRequest();
        String path = req.getPath().value();

        // 1. X-User-Id 외부 입력 strip (스푸핑 방어)
        ServerHttpRequest stripped = req.mutate().headers(h -> h.remove("X-User-Id")).build();

        // 2. 인증 면제 endpoint 통과
        if (isPublicEndpoint(path)) {
            return chain.filter(exchange.mutate().request(stripped).build());
        }

        // 3. Authorization 헤더 추출
        String authHeader = req.getHeaders().getFirst(HttpHeaders.AUTHORIZATION);
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            return unauthorized(exchange, "UNAUTHORIZED", "Missing Authorization header");
        }
        String token = authHeader.substring(7);

        // 4. JWT 검증 (서명 + 만료 + iss + aud)
        Jws<Claims> claims;
        try {
            claims = jwtParser.parseClaimsJws(token);
        } catch (ExpiredJwtException e) {
            return unauthorized(exchange, "TOKEN_EXPIRED", "Access token expired");
        } catch (JwtException e) {
            return unauthorized(exchange, "UNAUTHORIZED", "Invalid token");
        }

        // 5. (선택) Access Token blacklist 조회 — v1.0 미적용 (단명이라)

        // 6. X-User-Id 주입
        String userId = claims.getBody().getSubject();
        ServerHttpRequest authed = stripped.mutate()
            .header("X-User-Id", userId)
            .header("X-User-Role", claims.getBody().get("role", String.class))
            .build();

        return chain.filter(exchange.mutate().request(authed).build());
    }
}
```

### 3.7 WebSocket 인증 (04번 § 10.2)

> Gateway는 WS handshake에 JWT 필터 적용 X (HTTP GET + Upgrade에는 Authorization 헤더가 없음 — 브라우저 WebSocket API 한계).
>
> BFF Aggregator가 STOMP CONNECT 첫 프레임의 `Authorization: Bearer ...` 헤더를 직접 검증.

```java
// BFF WebSocketSecurityConfig (개념)
@Configuration
public class StompAuthChannelInterceptor implements ChannelInterceptor {

    @Override
    public Message<?> preSend(Message<?> message, MessageChannel channel) {
        StompHeaderAccessor accessor = StompHeaderAccessor.wrap(message);
        if (StompCommand.CONNECT.equals(accessor.getCommand())) {
            String token = extractBearerToken(accessor);
            Jws<Claims> claims = jwtParser.parseClaimsJws(token);
            String userId = claims.getBody().getSubject();
            accessor.setUser(new UserPrincipal(userId));
            // 검증 실패 시 STOMP ERROR 프레임 발행 + WS 종료
        }
        return message;
    }
}
```

### 4.2 자원 소유자 검증 (수직 권한 상승 방어)

> **흔한 취약점:** "본인 journal만 수정 가능"인데 다른 사용자의 journal_id를 보내도 통과 → IDOR (Insecure Direct Object Reference).

**표준 패턴 (Bible Service Journal 예):**

```java
@Service
public class JournalService {

    @Transactional
    public Journal updateJournal(Long journalId, Long currentUserId, JournalUpdateDto dto) {
        Journal journal = journalRepository.findById(journalId)
            .orElseThrow(() -> new NotFoundException("JOURNAL_NOT_FOUND"));

        // ⭐ 자원 소유자 검증 — 본인 자원이 아니면 403
        if (!journal.getUserId().equals(currentUserId)) {
            throw new ForbiddenException("JOURNAL_NOT_OWNED");
        }

        journal.update(dto);
        return journal;
    }
}
```

**금지 패턴 (1차 HMS에서 발생):**

```java
// ❌ 자원 소유자 검증 없음 — IDOR 취약점
public Journal updateJournal(Long journalId, JournalUpdateDto dto) {
    Journal journal = journalRepository.findById(journalId).orElseThrow();
    journal.update(dto);
    return journal;
}
```

### 6.3 Rate Limiting (04번 § 2.10 정합)

| 대상 | 제한 | 식별자 | 비고 |
| --- | --- | --- | --- |

| Auth `/auth/login` | replenishRate=1, burstCapacity=5 | client IP | brute-force 방어 (04번 § 2.10 v1.2) |
| Auth `/auth/register` | 3 req/min | client IP | |

> **분 단위 정확 제한 v1.1:** Spring Cloud Gateway 내장 RedisRateLimiter는 초 단위 → Bucket4j 또는 Custom Lua 검토.

### 6.4 CORS

| 항목 | v1.0 | v1.1 운영 |
| --- | --- | --- |
| `Access-Control-Allow-Origin` | `*` (시연·개발) | 정확한 origin 명시 (`https://qtai.app`) |
| `Access-Control-Allow-Credentials` | `false` (쿠키 안 씀, JWT만) | `false` 유지 |
| `Access-Control-Allow-Methods` | GET, POST, PATCH, DELETE, OPTIONS | 동일 |
| `Access-Control-Allow-Headers` | Authorization, Content-Type, X-Request-Id, traceparent | 동일 |
| `Access-Control-Max-Age` | 3600 | 동일 |

> **`Access-Control-Allow-Origin: *` + `Allow-Credentials: false`는 v1.0에 OK** (브라우저 CSRF 차단됨). 운영 시 origin 명시는 추가 보호선.

### 7.1 비밀번호 (§ 3.2 참조)

| 항목 | v1.0 | v1.1 |

## 8. OWASP Top 10 대응

> 2021 OWASP Top 10 기준. 각 카테고리에 대한 본 시스템의 대응을 박제.

### 8.1 A01: Broken Access Control

**대응:**
- § 4.2 자원 소유자 검증 (IDOR 방어)
- § 4.4 `@PreAuthorize` Spring Security
- Gateway X-User-Id strip + JWT 검증 (외부 spoofing 차단 — § 3.6)

**머지 리뷰 체크리스트:**
- [ ] service 메서드가 user_id 파라미터를 받는가?
- [ ] 자원의 owner를 검증하는가?
- [ ] 응답에 다른 사용자의 데이터가 섞일 수 있는가?

### 8.2 A02: Cryptographic Failures

**대응:**
- 비밀번호 bcrypt cost=12 (§ 3.2)
- JWT RS256 (§ 3.3.1) — HS256 사용 안 함

- 평문 시크릿 저장 금지 (§ 5)
- Refresh Token DB는 SHA-256 해시 저장 (§ 3.3.3)

**금지 사항:**
- MD5, SHA-1 (취약)
- 자체 개발 암호화 알고리즘
- 평문 저장

### 8.3 A03: Injection

**대응:**
- Spring Data JPA (Prepared Statement 자동) — Native Query는 `@Query(value = "...", nativeQuery = true)` 사용 시에만 + 파라미터 바인딩 강제

- AI 프롬프트 인젝션 방어 — § 8.10 신규 위협
- HTML 출력 방어 — Flutter는 자동 escape, 서버는 JSON만 반환

**금지 패턴:**
```java
// ❌ SQL Injection 가능
@Query(value = "SELECT * FROM users WHERE email = '" + email + "'", nativeQuery = true)

// ✅ 파라미터 바인딩
@Query(value = "SELECT * FROM users WHERE email = :email", nativeQuery = true)
List<User> findByEmail(@Param("email") String email);

// ✅ JPA가 더 안전 — native query 자체를 안 씀
List<User> findByEmail(String email);  // method name query
```

## 9. 로그 마스킹·감사 로깅

### 9.1 Logback PII 마스킹 패턴

> 모든 service의 `logback-spring.xml`에 동일 패턴 적용. 강태오가 W1에 표준 작성 → 4 service에 동일 적용.

```xml
<!-- logback-spring.xml -->
<configuration>

    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder class="net.logstash.logback.encoder.LogstashEncoder">
            <customFields>{"service":"${spring.application.name}"}</customFields>
        </encoder>
    </appender>

    <!-- 마스킹 변환기 -->
    <conversionRule conversionWord="maskedMsg"
                    converterClass="com.qtai.common.logging.PiiMaskingConverter"/>

    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
    </root>
</configuration>
```

```java
// common/logging/PiiMaskingConverter.java (BaseEntity처럼 4 service에 소스 복사 — 03번 § 5)
public class PiiMaskingConverter extends MessageConverter {

    private static final List<Pattern> SENSITIVE = List.of(
        // JWT
        Pattern.compile("eyJ[A-Za-z0-9_-]{20,}\\.[A-Za-z0-9_-]{20,}\\.[A-Za-z0-9_-]{20,}"),
        // Bearer token
        Pattern.compile("(?i)bearer\\s+[A-Za-z0-9._-]{20,}"),
        // Email (남기되 마스킹)
        Pattern.compile("([a-zA-Z0-9._-]{1,3})[a-zA-Z0-9._-]*@([a-zA-Z0-9-]+\\.[a-zA-Z0-9-.]+)"),
        // 비밀번호 필드
        Pattern.compile("(?i)(password|passwd|pwd)[\"':=\\s]*[^\"\\s,}]{4,}"),
        // 신용카드 (4-4-4-4 패턴 — 우리는 안 받지만 방어선)
        Pattern.compile("\\b(?:\\d[ -]*?){13,16}\\b"),
        // SSN 형식 (한국 주민번호 X — 안 받지만 방어선)
        Pattern.compile("\\b\\d{6}-\\d{7}\\b"),
        // API Key 패턴 (DeepSeek, Google)
        Pattern.compile("sk-[A-Za-z0-9_-]{20,}"),
        Pattern.compile("GOCSPX-[A-Za-z0-9_-]{28}")
    );

    @Override
    public String convert(ILoggingEvent event) {
        String msg = super.convert(event);
        for (Pattern p : SENSITIVE) {
            msg = p.matcher(msg).replaceAll(match -> {
                String matched = match.group();
                if (matched.contains("@")) {
                    // email: 앞 1~3글자 + ***@domain
                    return match.group(1) + "***@" + match.group(2);
                }
                return "***MASKED***";
            });
        }
        return msg;
    }
}
```

> **금지 패턴 — 절대 로그 출력 금지:**
> - JWT token (서명 + payload + signature)
> - 비밀번호 평문·해시 (일부라도)
> - Authorization 헤더 본문
> - DeepSeek API key
> - Google OAuth client secret
> - 사용자가 입력한 묵상 본문 전체 (감사 로그는 길이만)

## 10. 사고 대응 (Incident Response)

### 10.1 사고 분류

| 등급 | 정의 | 예 | SLA |
| --- | --- | --- | --- |
| 🔴 P0 (Critical) | 시스템 침해, PII 대량 누출, 재정 손실 | JWT private key 유출, 전 사용자 비밀번호 평문 노출 | 즉시 (분 단위) |
| 🟡 P1 (High) | 일부 사용자 영향, 일부 데이터 노출 | 1명 사용자의 다른 사용자 journal 조회 가능, OAuth client secret 유출 | 1시간 |
| 🟢 P2 (Medium) | 보안 통제 약화, 잠재적 위험 | gitleaks pre-commit 우회 발견, NetworkPolicy 누락 | 24시간 |
| ⚪ P3 (Low) | 정보성, 비기능적 영향 | 의존성 CVE 7.0 미만 발견 | W1 작업 |

### 10.2 사고 대응 절차

```
[1] 탐지 (Detection)
    ├─ Prometheus alert
    ├─ 사용자 신고
    └─ 코드 리뷰·정기 audit

[2] 격리 (Containment)
    ├─ P0: 영향 받은 service 즉시 K8s에서 scale=0 (Helm rollback)
    ├─ P1: 영향 받은 endpoint 차단 (Gateway 라우트 disable)
    └─ P2: 추가 모니터링

[3] 원인 분석 (Root Cause Analysis)
    ├─ 로그·메트릭·trace 수집
    └─ 영향 범위 산정 (얼마나 많은 사용자, 얼마나 많은 데이터)

[4] 복구 (Recovery)
    ├─ 코드·설정 patch
    ├─ DB 복원 (영향 시)
    └─ 서비스 재가동

[5] 사후 보고 (Post-Mortem)
    ├─ 24시간 이내 강태오가 incident report 작성 (Notion)
    ├─ 영향 받은 사용자 통보 (P0/P1)
    └─ 재발 방지책 ADR 발행
```

### 10.3 JWT Private Key 유출 대응 (P0 시나리오)

```
T+0 분: 유출 인지 (gitleaks 늦게 탐지 또는 외부 신고)

T+5 분:
  └─ 강태오 + 이지윤 즉시 통화
  └─ 새 RSA 키 쌍 생성 (kid: emergency-{ts})
  └─ K8s Secret jwt-keys에 v2 추가 (v1은 유지 — 검증만)

T+10 분:
  └─ Gateway Auth에 새 key 발급 활성화 (FORCE_NEW_KEY 플래그)
  └─ 모든 Pod rolling restart (Helm upgrade)
  └─ 사용자는 자동 재로그인 안내 (Flutter 측 401 시 로그인 화면)

T+15 분:
  └─ Redis blacklist에 v1 key로 발급된 모든 활성 jti 등록
  └─ 또는 Gateway 임시 차단 (모든 v1 토큰 401)

  └─ 사용자 통보 (Slack/Email — 영향 범위 따라)
```

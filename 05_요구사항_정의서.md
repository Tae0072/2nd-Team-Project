## 1. 프로젝트 개요

### 1.1 한 줄 정의

> QT와 성경을 잘 모르는 사용자도 오늘의 본문을 쉽게 읽고, 한/영 성경과 **AI가 미리 생성한 해설·장면 시뮬레이터**를 함께 보며, 개인 묵상 기록을 남길 수 있도록 돕는 심플한 QT 보조 앱.

### 1.2 해결하는 문제

| #   | 대상                              | 페인 포인트                               | 해결 방식                                                          |
| --- | --------------------------------- | ----------------------------------------- | ------------------------------------------------------------------ |
| 1   | 신앙 입문자                       | 어디서부터 읽어야 할지 막막               | 오늘의 QT 본문 → 해설 → 시뮬레이터 → 묵상 기록 순서의 단계형 흐름  |
| 2   | 시간 부족 직장인                  | 두꺼운 주석책 펼칠 시간 없음              | 필요할 때만 본문 요약·해설을 선택 열람                             |
| 3   | 묵상 방법론 부재 신앙인           | 자기중심 해석에 빠지는 위험               | 객관적 큐티 A~D 가이드 + 사전 작성된 묵상 가이드 질문              |
| 4   | 기존 성경 AI 앱 사용자            | 답변 출처 확인이 어려움, 이단성 검증 부재 | **AI 사전 생성 해설** + 편집자 에이전트 검증 + 응답 하단 출처 표시 |
| 5   | 본문 장면이 잘 안 그려지는 사용자 | 텍스트만으로 상황을 상상하기 어려움       | 시뮬레이터 — 본문 옆 아이콘 클릭 시 해당 장면 애니메이션 팝업      |
| 6   | 꾸준히 묵상하고 싶은 사용자       | 내 묵상 흐름을 한눈에 보기 어려움         | 마이페이지 묵상 달력 — 날짜 클릭 시 해당일 기록 조회               |

### 1.3 백엔드 구성 (2026-05-14 확정 — v2.2)

> **단일 백엔드 `qtai-server` (Modular Monolith).** 도메인 간 코드 분리는 패키지 경계 + DTO + Interface로 강제한다. K8s/Kafka 분리는 v2에서 검토한다.

**도메인 패키지:**

```
qtai-server/
└── com.qtai/
    ├── gatewayauth/   ← 인증·JWT 발급/검증, Rate Limit (강태오 횡단)
    ├── bff/           ← UseCase 집계, 화면용 응답 조립
    ├── bible/         ← 성경 본문·QT 본문·해설(A/B/C 테이블) 적재/조회 (이지윤·이승욱·김지민)
    ├── ai/            ← DeepSeek 연동, 해설 사전 번역·재해석, 편집자 검증 (강상민 주도)
    ├── journal/       ← 묵상 노트 + 묵상 달력 (bible 도메인 내 모듈)
    └── simulator/     ← 성경 장면 사전 생성·저장 (김태혁)
```

**담당:**

| 도메인               | 담당자                            | 주요 역할                                                                      |
| -------------------- | --------------------------------- | ------------------------------------------------------------------------------ |
| gatewayauth          | 강태오                            | JWT 발급·검증, Google OAuth, Rate Limit, 라우팅                                |
| bff                  | 강태오                            | 화면 단위 UseCase, 병렬 호출 집계 (`CompletableFuture`)                        |
| bible                | 이지윤·이승욱·김지민              | 성경 JSON 적재, QT 본문 범위 스크래핑, 해설 A/B/C 테이블, 묵상 노트, 묵상 달력 |
| ai (해설 생성기)     | 강상민 (주도)                     | B 테이블(영어 원문) → AI 재해석·번역 → C 테이블 적재. A 테이블과 비교 검증     |
| ai (편집자 에이전트) | 강상민 (구축) + 이지윤 (1차 검증) | C 산출물·시뮬레이터 산출물의 이단성·페르소나 이탈 검증                         |
| simulator            | 김태혁                            | 본문 장면 애니메이션 사전 생성·저장·재사용                                     |
| Flutter 앱           | 김지민 (1차) → 바이블 3인 합류    | 모바일 UI + 관리자 페이지                                                      |
| 횡단 (Lead)          | 강태오                            | PR 자동 검증 스크립트, 컨벤션, 인프라, CI/CD, 코드 리뷰                        |

### 1.4 서비스 방향성

- 초심자도 사용할 수 있는 쉬운 QT 앱 ("초딩도 이해할 정도로 쉽게" — 강사님)
- 예쁜 디자인보다 과하지 않은 심플한 디자인
- 커뮤니티 중심이 아닌 개인 공간 중심
- 필요할 때만 선택적으로 공유하는 구조
- 성경 본문, 영어 본문, 해설, 시뮬레이터, 묵상 기록을 한 흐름으로 제공
- AI는 사용자가 직접 질문하는 챗봇이 아니라 **본문에 딸리는 콘텐츠(해설·시뮬레이터)를 미리 만들어 두는 도구**

### 1.5 대상 사용자와 문제

| 주요 사용자                              | 사용자 문제                                                      |
| ---------------------------------------- | ---------------------------------------------------------------- |
| QT를 처음 접하는 사용자                  | QT가 무엇인지 모르고 앱을 켰을 때 무엇부터 해야 할지 모름        |
| 성경 지식이 많지 않은 사용자             | 성경 본문과 해설·묵상·적용 같은 용어가 어렵게 느껴짐             |
| 개인적으로 묵상하고 기록하고 싶은 사용자 | 개인적인 생각을 공개적으로 공유하기 부담스러움                   |
| 커뮤니티 공유에 부담을 느끼는 사용자     | 활발한 피드·댓글·반응 중심 구조가 부담스러움                     |
| 한/영 성경을 함께 보고 싶은 사용자       | 본문이 덩어리 단위로 제공되면 절별 비교가 불편함                 |
| 본문 장면 상상이 어려운 사용자           | 텍스트만으로는 인물·배경·사건을 떠올리기 어려움                  |
| 꾸준히 묵상하고 싶은 사용자              | 내가 며칠째 묵상하고 있는지, 빠진 날이 언제인지 한눈에 보고 싶음 |

### 1.6 핵심 사용 흐름

1. 사용자가 앱을 처음 실행한다.
2. 튜토리얼에서 QT가 무엇인지 안내받는다.
3. 앱은 별도 홈 화면 없이 오늘의 QT 화면으로 바로 진입한다 (오늘 QT 데이터 먼저 로딩, 나머지 성경 데이터는 백그라운드 로딩).
4. 오늘의 성경 본문을 절별 한글/영어 형태로 읽는다.
5. 어려운 단어, 본문 요약, **AI가 사전 생성하고 편집자 에이전트가 검증한 해설(C 테이블)** 을 선택 열람한다.
6. 본문 옆 시뮬레이터 아이콘을 눌러 **사전 생성된 장면 애니메이션**을 열람할 수 있다.
7. 개인 묵상 기록을 4개 섹션(느낀 점, 기억할 구절, 적용할 점, 기도)으로 작성한다 (자동 저장).
8. **운영자가 사전 큐레이션한 찬양 리스트**에서 마음에 드는 곡을 내 찬양 목록에 저장하거나 제거한다.
9. 기록은 기본적으로 비공개 저장된다.
10. 사용자가 선택한 경우에만 익명 형태로 공유한다.
11. 마이페이지에서 **묵상 달력**으로 내 묵상 이력을 확인하고, 날짜 클릭 시 해당일 기록을 조회한다.

> **사용자는 AI에게 직접 질문하지 않는다.** 사용자가 보는 모든 AI 콘텐츠(해설·시뮬레이터)는 사전에 생성되어 DB에 저장된 것을 조회하는 방식이다.
> **AI는 찬양을 추천하지 않는다.** 찬양은 운영자가 사전 큐레이션한 리스트에서 사용자가 선택한다.

### 2.2 F-01: 성경 묵상 화면 (입체 뷰)

**설명:** 사용자가 성경 구절을 선택하면 한글 성경·영어 성경을 절별로 비교하고, 필요할 때 본문 요약과 해설을 열람할 수 있는 화면.

**사용자 흐름:**

1. 앱 실행 → 오늘의 QT 화면으로 바로 진입하거나, 일반 성경 보기에서 책·장·절 선택
2. 입체 묵상 화면 진입
3. 한/영 성경이 절별로 표시되고 긴 본문은 스크롤 동기화 (Sync Scroll)
4. 사용자가 필요할 때 본문 요약·해설 패널을 선택 열람 (해설은 C 테이블 — AI 재해석본)
5. 묵상 기록·시뮬레이터 액션은 오늘의 QT 본문에서만 제공한다. 일반 성경 보기의 자유 본문은 조회 전용이다.

### 2.3 F-02: AI 콘텐츠 사전 생성 파이프라인 (백엔드)

> **v2.1 변경:** 사용자 ↔ AI 실시간 Q&A 기능은 전면 제거되었다. 본 기능은 사용자에게 직접 노출되지 않는 **백엔드 콘텐츠 생성 파이프라인**이다.
> **v2.3 보정:** 주석 데이터는 **A/B/C 3-테이블 구조**로 관리한다. B 테이블 원문을 AI로 재해석해 C 테이블을 생성함으로써 원문 직접 사용·표절 위험을 줄이고 저작권 리스크를 낮춘다.

**설명:** AI는 오늘 QT 본문에 대한 **해설**과 **시뮬레이터 클립**을 사전에 생성·검증한 뒤 DB에 적재한다. 사용자는 산출물을 조회만 한다.

#### 2.3.1 해설 3-테이블 파이프라인 (v2.2 신규)

| 테이블                          | 역할                                | 사용자 노출      | 소스                        | 비고                                                      |
| ------------------------------- | ----------------------------------- | ---------------- | --------------------------- | --------------------------------------------------------- |
| **A** (`commentary_modern_ref`) | 최신 한국어 주석 — **검증 참조용**  | ❌ 비공개        | 최신 주석 자료 1종          | C 산출물의 정확도·정합성 비교 검사에만 사용. UI 노출 금지 |

**AI 배치 운영 정책:**

AI 배치는 매일 **04:00 KST QT 본문 범위 수집 이후** 실행한다. 사용자 요청 경로에서는 LLM을 호출하지 않는다.

1. 04:00 KST QT 본문 범위 수집
2. 자체 성경 DB에서 한/영 본문 조회
3. B 테이블 영어 주석 조회
4. DeepSeek으로 한국어 해설 초안 생성
5. 편집자 에이전트 검증
6. 통과 시 C 테이블 저장
7. 시뮬레이터 생성 또는 상태 갱신
8. 오늘 QT 캐시 교체
9. `AiContentGeneratedEvent` 발행

**재시도·실패 처리:**

| 단계 | 재시도 | 실패 시 처리 |
| --- | --- | --- |
| QT 본문 범위 수집 | 최대 3회 | 전날 QT 캐시 유지, 관리자 알림 |
| DeepSeek 해설 생성 | 최대 2회 | `ai_runs`에 `FAILED` 기록 |
| 편집자 검증 | 최대 2회 | `REJECTED` 또는 `FAILED` 기록 |
| C 테이블 저장 | 최대 3회 | 트랜잭션 롤백, 실패 로그 |
| 시뮬레이터 생성 | 최대 1회 | `MISSING` 또는 `FAILED` 상태 저장 |
| 캐시 교체 | 최대 3회 | 기존 캐시 유지 |

**AI 실행 이력 (`ai_runs`) 최소 컬럼:**

| 컬럼 | 설명 |
| --- | --- |
| `id` | 실행 ID |
| `qtId` | 오늘 QT 식별자 |
| `runType` | `EXPLANATION` / `SIMULATOR` / `EDITOR_VERIFICATION` |
| `status` | `PENDING` / `RUNNING` / `SUCCEEDED` / `FAILED` / `REJECTED` |
| `modelProvider`, `modelName` | 모델 제공자와 모델명 |
| `promptVersion` | 프롬프트 버전 |
| `inputHash`, `outputHash` | 입력·출력 중복 검증용 해시 |
| `retryCount` | 재시도 횟수 |
| `errorMessage` | 실패 사유 |
| `startedAt`, `finishedAt` | 실행 시작·종료 시각 |

동일한 `qtId + runType + promptVersion + inputHash` 조합은 중복 실행하지 않는다. 이미 `SUCCEEDED` 상태인 산출물이 있으면 재사용한다. 관리자 재생성 트리거는 기존 산출물을 덮어쓰지 않고 새 version으로 저장한다.

**프롬프트 버전 관리:**

| 프롬프트 | 예시 버전 |
| --- | --- |
| 해설 생성기 | `explanation-generator-v1` |
| 편집자 검증기 | `editor-verifier-v1` |
| 시뮬레이터 장면 생성 | `simulator-scene-v1` |

C 테이블과 검증 로그에는 사용된 `promptVersion`, `modelName`, `sources`를 함께 저장한다.

**편집자 에이전트 검증 결과:**

| 결과 | 의미 |
| --- | --- |
| `PASS` | 사용자 노출 가능 |
| `REJECT` | 사용자 노출 금지, 재생성 필요 |
| `NEEDS_REVIEW` | 관리자 또는 담당자 확인 필요 |

검증 항목은 본문 범위와 해설 내용의 일치 여부, B 테이블 원문 과다 직역/복제 여부, A 테이블 검증 기준과의 사실 충돌 여부, 단정적 설교·신앙 평가 표현 포함 여부, 초심자 이해 가능 여부를 포함한다.

### 2.4 F-03: 묵상 노트 관리

**설명:** 사용자가 오늘의 QT 본문을 읽고 느낀 점, 기억할 구절, 적용할 점, 기도 제목을 분리해 저장한다. 기록은 기본 비공개이며, 사용자가 원할 때만 나눔으로 공개한다. AI는 묵상 노트에 직접 관여하지 않는다.

> **구현 기준:** 별도 Journal 서비스/도메인 패키지를 만들지 않는다. 묵상 노트는 `com.qtai.bible` 도메인 내부의 `journal` 하위 모듈로 운영한다.

**사용자 흐름:**

1. 오늘의 QT 화면에서 내 생각 기록 영역에 진입
2. bible 도메인이 오늘 날짜 기준 DRAFT 상태의 묵상 노트를 멱등 생성 또는 기존 DRAFT 조회
3. 사용자가 4개 섹션(느낀 점, 기억할 구절, 적용할 점, 기도)을 작성
4. 입력 내용은 별도 저장 버튼 없이 자동 저장
5. 마이페이지 → 묵상 달력 또는 묵상 기록 리스트 확인
6. 노트 클릭 → 상세 보기 (구절·날짜·개인 기록)
7. 기록 수정, 삭제, 나눔 공개 여부 선택

**상세 요구사항:**

| ID      | 요구사항                                                                                                | 비고                                           |
| ------- | ------------------------------------------------------------------------------------------------------- | ---------------------------------------------- |
| F-03-01 | 묵상 노트는 오늘 QT 기준 `POST /api/v1/journals/today`로 멱등 생성/조회한다                             | 자유 본문용 수동 생성 API 없음                 |
| F-03-02 | 묵상 노트 목록을 조회한다 (`GET /api/v1/journals`)                                                      |                                                |

## 5. 화면 요구사항

### 5.1 온보딩 화면

- "QT가 처음이신가요?" 문구 제공
- QT의 의미와 앱 사용 순서 안내
- 오늘의 QT 시작 버튼 제공

### 5.2 초기 진입 (별도 홈 화면 없음)

- MVP 첫 진입은 별도 홈 화면 없이 오늘의 QT 화면으로 바로 진입
- 오늘 QT 데이터 먼저 로딩 후 화면 진입 (YouTube식 점진 로딩)
- 나머지 성경 데이터는 백그라운드 로딩
- 로그인은 첫 진입에서 강제하지 않고, 기록 저장·해설·찬양 저장·묵상 달력 기능 진입 시 유도

### 5.3 오늘의 QT 화면

- 오늘의 성경 본문 (성서 유니온 00:00 KST 공개 / 우리 시스템 04:00 KST 수집 기준 본문 범위)
- 절별 한글/영어 보기 (GitHub JSON 적재 데이터)
- 본문 한 줄 요약 선택 열람 (C 테이블)
- 어려운 단어 설명 선택 열람 (C 테이블)
- 해설 보기 선택 열람 (C 테이블 — AI 재해석 + 편집자 검증 완료본)
- 시뮬레이터 아이콘 (상태가 `READY`이면 활성화, `MISSING`/`FAILED`/`DISABLED`이면 비활성화)
- 묵상 기록 자동 저장 (느낀 점, 기억할 구절, 적용할 점, 기도)
- 운영자 사전 큐레이션 찬양 리스트에서 곡 선택 → 내 찬양 목록 저장/제거

> AI 질문하기 버튼·AI 추천 찬양 영역은 두지 않는다.

### 5.4 마이페이지

- **묵상 달력 (F-13)** — 월별 달력, 묵상한 날 체크 표시, 날짜 클릭 시 해당 기록 이동
- 묵상 기록 리스트 (날짜 역순)
- 내 찬양 목록 (저장한 큐레이션 곡)
- 공개 여부 선택
- 사용자 입력 글자 수 제한 없음 (D형 묵상도 가능)
- 기본값 비공개, 저장 후 사용자가 원할 때만 나눔으로 공개

### 5.5 묵상 기록 상세 화면

- 해당일 본문·날짜·개인 기록(4섹션)
- 수정·삭제·공개 여부 토글
- 마이페이지의 묵상 달력에서 진입

### 5.6 익명 공유 화면

- 기본값은 비공개
- 사용자가 선택한 기록만 공유
- 작성자 정보는 노출하지 않음
- 조용히 읽는 블로그형 구조 지향
- 좋아요, 댓글, 신고의 최소 기능 제공
- 팔로우, 실시간 피드, 랭킹 중심 구조 제외

### 5.7 시뮬레이터 팝업

- 본문 옆 아이콘 클릭 → 사전 생성된 짧은 애니메이션 클립 재생
- 정지/재생/닫기 컨트롤
- `MISSING`/`FAILED`/`DISABLED` 상태는 아이콘 비활성화

## 7. 인프로세스 이벤트 요약 (Kafka 보류)

| 이벤트 (인프로세스)          | 발행 도메인 | 구독 도메인                  | 목적                                                  |
| ---------------------------- | ----------- | ---------------------------- | ----------------------------------------------------- |
| `UserActivityTrackedEvent`   | bff         | bible                        | 구절 조회 이력 기록                                   |
| `AiContentGeneratedEvent`    | ai          | bible, bff, 관리자           | AI 해설(C)·시뮬레이터 산출물이 DB에 적재되었음을 알림 |
| `EditorRejectedEvent`        | ai (편집자) | 관리자                       | 편집자 에이전트가 산출물을 반려했을 때 발행           |
| `JournalCreatedEvent`        | bible       | 다수 + 묵상 달력 캐시 무효화 | 노트 생성 완료 알림                                   |
| `JournalUpdatedEvent`        | bible       | 다수 + 묵상 달력 캐시 무효화 | 노트 수정 완료 알림                                   |
| `JournalDeletedEvent`        | bible       | 다수 + 묵상 달력 캐시 무효화 | 노트 삭제 완료 알림                                   |
| `NotificationRequestedEvent` | 다수        | bff                          | 인앱 알림 큐잉                                        |

> Spring `ApplicationEventPublisher`는 Kafka처럼 영속 메시징을 보장하지 않는다. v1은 이벤트 유실률 0%를 목표로 두지 않고, **핸들러 실패 로그 + 재처리 가능**을 목표로 한다. 이벤트 핸들러 실패 시 `eventId`와 error log를 기록하며, 재처리가 필요한 이벤트는 DB 상태 기준으로 관리자 또는 배치가 재시도할 수 있어야 한다. 묵상 노트처럼 이력 보존이 필요한 기능은 별도 journal event table에 영속 기록한다.

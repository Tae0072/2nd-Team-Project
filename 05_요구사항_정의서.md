## 1. 프로젝트 개요

### 1.1 한 줄 정의

> 성경 구절을 클릭하면 한/영 성경과 주석이 동시에 보이고, AI가 큐티(QT) A~D형 패턴으로 묵상을 코칭해주는 Flutter 모바일 앱.

### 1.2 해결하는 문제

| # | 대상 | 페인 포인트 | 해결 방식 |
| --- | --- | --- | --- |
| 1 | 신앙 입문자 | 어디서부터 읽어야 할지 막막 | 큐티 A형(관찰)부터 단계별 AI 코칭 |
| 2 | 시간 부족 직장인 | 두꺼운 주석책 펼칠 시간 없음 | 구절 클릭 → 주석 자동 노출 |
| 3 | 묵상 방법론 부재 신앙인 | 자기중심 해석에 빠지는 위험 | 객관적 큐티 A~D 가이드 + 꼬리 질문 |
| 4 | 기존 성경 AI 앱 사용자 | 단순 단답형 한계 | 코칭형(소크라테스식) 대화 + SSE 스트리밍 |

### 1.3 서비스 구성 (2026-05-12 확정)

| 서비스 | 담당자 | 주요 역할 |
| --- | --- | --- |
| API Gateway | 강태오 | JWT 검증, 라우팅, Rate Limit |
| BFF Aggregator | 강태오 | 병렬 호출 집계, UseCase 패턴 |
| Bible Service | 이지윤·이승욱 | 성경·주석·QT·묵상일지(Journal) 통합 |
| AI Service | 강태오(팀장)·김태혁·강상민 | DeepSeek 코칭 대화, ChromaDB RAG, SSE |
| Flutter 앱 | 김지민 | 모바일 UI, 관리자 웹 |

> **Auth Service 제거:** JWT 발급·검증은 Gateway에서 처리.
> **Journal Service 제거:** 묵상일지 기능 Bible Service로 통합.

### 2.2 F-01: 성경 묵상 화면 (입체 뷰)

**설명:** 사용자가 성경 구절을 선택하면 한글 성경·영어 성경·주석을 동시에 볼 수 있는 화면.

**사용자 흐름:**
1. 앱 실행 → 성경 목록에서 책·장·절 선택
2. 입체 묵상 화면 진입
3. 한/영 성경이 나란히 표시, 스크롤 동기화 (Sync Scroll)
4. 하단 또는 사이드에 주석 패널 자동 노출
5. 구절 선택 후 AI 대화로 이동 가능

**상세 요구사항:**

| ID | 요구사항 | 비고 |
| --- | --- | --- |
| F-01-01 | 한글 성경(개역한글) + 영어 성경(KJV)을 동시에 표시한다 | 두 성경 스크롤 동기화 |
| F-01-02 | 주석(Matthew Henry 영문, 더미 한글)을 함께 표시한다 | |
| F-01-03 | BFF가 Bible Service에 한글·영어·주석을 병렬 호출하여 단일 응답으로 반환한다 | CompletableFuture 사용 |
| F-01-04 | 자주 조회되는 구절은 Redis에 캐시한다 | TTL 24시간, 인기 구절 Top 100은 7일 |
| F-01-05 | 구절 조회 이력은 Kafka(`user.activity.tracked`)로 비동기 기록한다 | |
| F-01-06 | 각 서비스 호출 타임아웃은 800ms, 실패 시 부분 응답을 반환한다 | 폴백 처리 |
| F-01-07 | 성경 목록 조회(`GET /bible/books`)를 제공한다 | |

**저작권 제약:**

| 데이터 | 사용 가능 여부 |
| --- | --- |
| KJV (영어 성경) | ✅ Public Domain |
| 개역한글 | ⚠️ 비상업·교육 목적 한정, 출처 표기 필수 |
| Matthew Henry 주석 (영문) | ✅ Public Domain |
| 개역개정 / ESV / NIV | ❌ **사용 금지** |
| 한글 주석 / 신학 논문 | ❌ 더미 데이터로 대체 |

### 2.3 F-02: AI 코칭 대화 (큐티 A~D형)

**설명:** 사용자가 성경 구절을 선택한 후 AI와 대화하며 큐티(QT) 묵상을 진행한다. AI는 소크라테스식 꼬리 질문으로 단계별 코칭을 제공한다.

**큐티 단계 정의:**

| 단계 | 이름 | 설명 |
| --- | --- | --- |
| A | 관찰 | "본문에서 무엇이 보이나요?" — 사실 파악 |
| B | 해석 | "이 구절이 왜 쓰였을까요?" — 맥락 이해 |
| C | 적용 | "내 삶에 어떻게 적용할 수 있을까요?" |
| D | 결단 | "오늘 한 가지 실천하겠습니다" (MVP 시간 가능 시 구현) |

**사용자 흐름:**
1. 구절 선택 후 "AI와 묵상하기" 버튼 클릭
2. AI 세션 생성 (`POST /ai/sessions`)
3. AI가 A형(관찰) 질문으로 시작
4. 사용자 답변 입력 (`POST /ai/sessions/{id}/turns`)
5. AI 응답을 SSE 스트리밍으로 실시간 표시
6. 단계 완료 후 다음 단계로 자동 진행
7. 대화 종료 → 묵상 노트 자동 생성

**상세 요구사항:**

| ID | 요구사항 | 비고 |
| --- | --- | --- |
| F-02-01 | AI 응답을 SSE(Server-Sent Events)로 스트리밍한다 | Spring SseEmitter |
| F-02-02 | SSE 이벤트 순서는 `turn_started → token → rag_sources → turn_completed → [DONE]`이다 | |
| F-02-03 | ChromaDB에서 관련 주석·논문 임베딩을 RAG로 조회한다 | |
| F-02-04 | RAG 컨텍스트 외 추론을 금지하고 출처 인용을 강제한다 | 신학적 오류 방지 |
| F-02-05 | LLM은 DeepSeek API (OpenAI 호환)를 사용한다 | Spring RestClient |
| F-02-06 | 첫 토큰 응답 타임아웃은 5초, 초과 시 안내 메시지를 반환한다 | |
| F-02-07 | AI 세션·대화 이력은 MySQL(`AI_SESSIONS`, `AI_TURNS`)에 저장한다 | |
| F-02-08 | 대화 종료 시 Kafka(`ai.session.completed`)를 발행한다 | Bible Service가 구독 → 묵상 노트 자동 생성 |
| F-02-09 | 큐티 A~C형은 MVP에 반드시 포함한다. D형은 시간 가능 시 추가한다 | |
| F-02-10 | AI API Key는 K8s Secret으로만 관리하며 코드·YAML에 평문 저장을 금지한다 | |

### 2.4 F-03: 묵상 노트 관리

**설명:** AI 대화 종료 후 자동으로 생성되는 묵상 노트를 조회·수정·삭제한다.

**사용자 흐름:**
1. AI 대화 종료 → 자동으로 DRAFT 상태의 묵상 노트 생성
2. 마이페이지 → 묵상 기록 리스트 확인
3. 노트 클릭 → 상세 보기 (구절·날짜·AI 대화 요약·개인 메모)
4. 메모 추가·수정 후 저장

**상세 요구사항:**

| ID | 요구사항 | 비고 |
| --- | --- | --- |
| F-03-01 | 묵상 노트는 `ai.session.completed` Kafka 이벤트를 Bible Service가 구독하여 자동 생성한다 | 수동 생성 API(`POST /api/v1/journals`) 없음 |
| F-03-02 | 묵상 노트 목록을 조회한다 (`GET /api/v1/journals`) | |
| F-03-03 | 묵상 노트 단건을 조회한다 (`GET /api/v1/journals/{id}`) | |
| F-03-04 | 묵상 노트를 수정한다 (`PATCH /api/v1/journals/{id}`) | 개인 메모 추가·수정 |
| F-03-05 | 묵상 노트를 삭제한다 (`DELETE /api/v1/journals/{id}`) | |
| F-03-06 | 이벤트 변경 이력을 조회한다 (`GET /api/v1/journals/{id}/events`) | 이벤트 소싱 |
| F-03-07 | 묵상 노트 생성·수정·삭제 이벤트는 Kafka로 발행한다 | producer: bible-service |
| F-03-08 | Kafka 컨슈머는 멱등성 키를 검증하여 중복 처리를 방지한다 | |

### 2.5 F-04: 회원 인증

**설명:** JWT 기반 인증으로 앱에 로그인한다. 인증 처리는 Gateway에서 담당한다.

**상세 요구사항:**

| ID | 요구사항 | 비고 |
| --- | --- | --- |
| F-04-01 | Google 소셜 로그인을 지원한다 | OAuth 2.0 |
| F-04-02 | Access Token 유효기간은 30분(1800s)이다 | RS256 알고리즘 |
| F-04-03 | Refresh Token 유효기간은 14일이다 | |
| F-04-04 | Refresh Token 블랙리스트는 Redis에 저장한다 | `auth:refresh:revoked:{jti}` |
| F-04-05 | JWT 검증은 Gateway 필터에서 처리한다 | Auth Service 없음 |
| F-04-06 | 계정 탈퇴 기능은 MVP 범위에서 제외한다 | v1.1에서 구현 |

## 6. Kafka 이벤트 요약

| 토픽 | 발행 서비스 | 구독 서비스 | 목적 |
| --- | --- | --- | --- |

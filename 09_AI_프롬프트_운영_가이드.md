## 1. 개요·범위·1차 패턴 분석

### 1.1 본 문서가 다루는 범위

본 문서는 QT-AI의 **AI Service (`ai-service/`)** 에서 일어나는 모든 LLM 호출·RAG·프롬프트 운영의 표준을 박제한다. 강상민이 단독으로 RAG·SSE·신학 가드레일·프롬프트 인젝션 방어·Anthropic 통합을 모두 구현하므로, 이 한 곳이 무너지면 시연 전체가 무너진다.

본 문서가 **다루는 것:**
- Anthropic Claude API 호출 표준 (모델·파라미터·timeout)
- 큐티 4단계(A→B→C→D) 시스템 프롬프트 4개 박제 (수정 시 ADR 필수)
- RAG: ChromaDB collection 구성·임베딩 모델·검색 파라미터·context window 한도
- 신학 가드레일: 정통 신학 범위 + 회피 영역 (이단·분파·정치·예언적 발화 금지)
- 프롬프트 인젝션 방어 3 계층 (입력 sanitize → system role 분리 → 출력 검증)
- 출력 검증: RAG 출처 인용 강제·존재하지 않는 성경 구절 환각 차단
- SSE 스트림 4 이벤트 (token·turn_completed·error·end) 발행 표준
- LLM 비용·rate limit·timeout 정책 (60/min/user, 4분 타임아웃)
- 평가 셋: golden-set 100건·injection-set 30건·theology-set 50건
- 운영 모니터링 (드리프트 감지·CSAT)

본 문서가 **다루지 않는 것 (다른 문서 참조):**
- API 계약 (`/ai/sessions/...` 엔드포인트 시그니처) → 04번 § 6
- SSE 프로토콜 와이어 포맷 → 04번 § 6.3
- AI Service 보안 (시크릿 4계층 방어) → 05번 § 4
- AI Service 배포·환경 변수 → 06번 § 5
- AI Service 테스트 (단위·통합·E2E 게이트) → 07번 § 8
- Flutter 클라이언트의 SSE 수신 → 08번 § 8

## 2. 큐티 4단계 정의 + 사용자 흐름

### 2.1 큐티 4단계 — 도메인 정의

QT-AI의 AI 코칭은 **귀납적 성경 묵상법 (Inductive Bible Study)** 의 4단계를 따른다. 이는 한국 개신교 큐티 도서·세미나에서 표준으로 사용되는 패턴.

| 단계 | 영문 | 한글 | LLM 역할 | 예상 turn 수 |
| --- | --- | --- | --- | --- |
| **A. 관찰** | Observation | 본문에 무엇이 적혀 있는가? | 본문 안의 사실(인물·시간·장소·동작·반복) 발견을 코칭. 답을 직접 주지 않고 사용자가 보게 함 | 2~4 turn |
| **B. 해석** | Interpretation | 그 본문은 무엇을 의미하는가? | 본문의 시대·문맥·언어 배경 활용한 의미 해석을 코칭. RAG로 주석·논문 인용 | 3~5 turn |
| **C. 적용** | Application | 그 의미는 내 삶에 어떻게 적용되는가? | 사용자의 삶 상황에 본문 의미를 연결하는 질문 | 2~4 turn |
| **D. 결단** | Decision | 오늘 무엇을 결심하고 행동할 것인가? | 구체적 행동 1개를 사용자가 정하도록 코칭. LLM이 직접 명령 X | 1~2 turn |

**총 예상 turn:** 8~15 turn / 세션. 평균 11 turn으로 설계 (§ 14 비용 베이스라인).

### 2.2 단계 진행 규칙 — A→B→C→D 단방향

| 규칙 | 적용 |
| --- | --- |
| 단계 역행 금지 | A→B 진행 후 A로 못 돌아감 (사용자 요청해도 차단). 단, 세션 abandon 후 새 세션은 가능 |
| 단계 건너뜀 금지 | A→C 직접 진행 X. 반드시 A→B→C→D 순서 |
| 자동 진행 vs 수동 진행 | LLM이 "A 단계 충분히 다뤘습니다, B로 넘어갈까요?" 제안 → 사용자 확인 후 BFF가 step 변경 |
| D 완료 = 세션 종료 | D 단계 마지막 turn의 `turn_completed` 이벤트 후 SSE `end` + 세션 status `COMPLETED` |
| 중간 abandon | 어느 단계에서나 사용자가 `/ai/sessions/{id}/abandon` 호출 가능 → status `ABANDONED` |

### 2.3 사용자 흐름 (시퀀스)

```
1. 사용자 → Flutter PassageViewPage (입체 묵상 화면)
   "AI 큐티 시작" 버튼 클릭
2. Flutter → BFF: POST /ai/sessions { bookCode, chapter, verse }
3. BFF → AI Service: POST /ai/sessions { userId, bookCode, ... }
4. AI Service:
   a. AI_SESSIONS row INSERT (status=ACTIVE, current_step=A)
   b. PROMPT_TEMPLATES.find(step=A, version=current) → systemPromptA
   c. RAG: ChromaDB 검색 (passage_context query) → top-K 5
   d. 응답: { sessionId, currentStep: 'A' }
5. Flutter → AiSessionPage 진입
6. 사용자 입력 "이 구절에서 누가 등장하나요?"
7. Flutter → BFF: POST /ai/sessions/{id}/turns SSE { step: 'A', content }
8. BFF → AI Service: SSE proxy
9. AI Service:
   a. 입력 sanitize (§ 7.1)
   b. user_message + system_prompt_A + RAG context 조립
   c. Anthropic Claude API 호출 (stream=true)
   d. token 단위 SSE 발행 (event: token)
   e. 응답 완료 시 출력 검증 (§ 8) + RAG 출처 인용 추출
   f. SSE 발행 (event: turn_completed { turnId, ragSources })
   g. AI_TURNS row INSERT
   h. SSE 발행 (event: end)
10. (반복) A 단계 충분 시 LLM이 "B로 넘어갈까요?" 제안
11. 사용자 동의 → Flutter → POST /api/v1/ai/sessions/{id}/turns (사용자 응답 제출, 단계 전환 자동)
12. AI Service: AI_SESSIONS.current_step = B
13. (반복 B → C → D)
14. D 마지막 turn 완료 시:
    - AI_SESSIONS.status = COMPLETED
    - Kafka 발행: ai.session.completed { sessionId, userId, summary }
    - Journal Service consume → JOURNAL 자동 생성 (DRAFT)
    - BFF Aggregator → Flutter STOMP 푸시 (notification)
```

### 2.4 단계별 RAG 사용 차등

각 단계에서 RAG 컨텍스트 사용량이 다름. 비용·관련성 균형:

| 단계 | RAG top-K | RAG 사용 깊이 | 근거 |
| --- | --- | --- | --- |
| A. 관찰 | 0 (사용 X) | RAG 없이 본문만 | 관찰은 본문 내부 사실 발견 — 외부 주석 불필요 |
| B. 해석 | 5 (가장 많음) | 주석·논문 적극 인용 | 의미 해석은 권위 있는 출처 필수 |
| C. 적용 | 2 | 적용 사례 1~2개 | 너무 많은 사례는 오히려 사용자 결단 방해 |
| D. 결단 | 0 (사용 X) | RAG 없이 사용자 자력 | 결단은 사용자 본인이 — LLM이 명령 X |

> **W1 Lock-in:** 단계별 RAG 사용 차등은 본 문서가 정의 — `ai-service/src/main/resources/rag/strategy.yaml`에 박제 (§ 5.3).

### 2.5 1차(HMS) ↔ 큐티 4단계 매핑 분석

1차 HMS는 의료 예약 시스템 — 도메인 다름. 그러나 **사용자 워크플로 단계 진행 패턴**은 매핑 가능:

| 1차 (HMS) | 2차 (QT-AI) | 공통 패턴 |
| --- | --- | --- |
| 예약 단계 (slot 선택 → 환자 정보 → 결제 → 확정) | 큐티 단계 (A → B → C → D) | 단방향 진행 + 중간 abandon 가능 |
| 단계 역행 금지 (결제 후 slot 변경 X) | 단계 역행 금지 (B 후 A 못 돌아감) | 데이터 무결성 보호 |
| 단계별 검증 룰 | 단계별 LLM 시스템 프롬프트 | 단계 의미 명확화 |
| 자동 진행 트리거 (의사 승인 시 결제 가능) | LLM 제안 → 사용자 동의 (B로 진행) | 단계 전환 명시적 |

→ HMS의 "예약 단계 진행 룰 명세 부족 → 결제 후 slot 변경 시도로 race" 사고가 본 시스템에서는 "B 단계 도중 A로 돌아감 시도 → race로 turn 순서 깨짐" 형태로 재발할 가능성. 가드레일: AI_SESSIONS.current_step의 enum CHECK 제약 + step 전환 메서드 1곳에서만 (§ 9.5).

### 3.1 모델 결정

| 항목 | 결정 | 대안 / 기각 사유 |
| --- | --- | --- |
| 제공자 | **Anthropic** | OpenAI — 응답 톤이 큐티 도메인에 덜 맞음 (실험 결과) / 자체 모델 — 6주 일정 불가 |
| 모델 | **Claude Sonnet 4.5** (`claude-sonnet-4-5`) | Opus — 비용 4배·시연 부담 / Haiku — 신학 가드레일 정확도 부족 |
| API 모드 | **Messages API** (`/v1/messages`) | Completions — deprecated |

> **W0 Lock-in:** 모델 변경 시 ADR + golden-set 100건 회귀 검증 필수. `claude-sonnet-4-5` 외 사용 X.

### 3.2 호출 파라미터 표준 (단계별)

각 단계별로 다른 파라미터. **temperature는 단계 의도에 맞춰 설정**:

| 단계 | model | max_tokens | temperature | top_p | stop_sequences |
| --- | --- | --- | --- | --- | --- |
| A. 관찰 | claude-sonnet-4-5 | 800 | 0.3 (사실 관찰은 결정적) | 1.0 | `["\n\n사용자:"]` |
| B. 해석 | claude-sonnet-4-5 | 1200 | 0.5 (해석은 약간 다양성) | 1.0 | `["\n\n사용자:"]` |
| C. 적용 | claude-sonnet-4-5 | 1000 | 0.7 (적용은 창의적 질문) | 1.0 | `["\n\n사용자:"]` |
| D. 결단 | claude-sonnet-4-5 | 600 | 0.4 (결단은 명확하게) | 1.0 | `["\n\n사용자:"]` |

**근거 — temperature 차등:**
- 관찰(A) 0.3: 본문 내부 사실은 누가 봐도 같아야 함 (Determinism)
- 해석(B) 0.5: 신학적 해석은 다양한 관점 존중하되 정통 범위 안에서
- 적용(C) 0.7: 사용자 삶 상황에 맞는 다양한 질문 생성
- 결단(D) 0.4: 사용자가 결정할 행동 1개에 집중되도록

`max_tokens` 1200 한도 (B 해석 단계). 1 turn 평균 비용:
- 입력 (system + RAG + history): ~3,500 token (4,500 token max)
- 출력: ~800~1,200 token

세션 평균 11 turn → 입력 누적 ~38,500 token + 출력 ~11,000 token = 약 USD 0.20/세션 (Sonnet 4.5 가격 기준).

### 3.3 호출 코드 표준

```java
// ai-service/src/main/java/com/qtai/ai/anthropic/AnthropicClient.java
@Component
public class AnthropicClient {
    private final WebClient webClient;
    private final String apiKey;  // 환경 변수 ANTHROPIC_API_KEY

    public AnthropicClient(@Value("${anthropic.api.key}") String apiKey) {
        this.apiKey = apiKey;
        this.webClient = WebClient.builder()
            .baseUrl("https://api.anthropic.com/v1")
            .defaultHeader("x-api-key", apiKey)
            .defaultHeader("anthropic-version", "2023-06-01")
            .defaultHeader("content-type", "application/json")
            .build();
    }

    public Flux<AnthropicEvent> streamMessage(MessageRequest req) {
        return webClient.post()
            .uri("/messages")
            .bodyValue(req)
            .retrieve()
            .bodyToFlux(ServerSentEvent.class)
            .map(this::toAnthropicEvent)
            .timeout(Duration.ofMinutes(4))  // SSE 4분 타임아웃 (§ 10.3)
            .doOnError(e -> log.warn("Anthropic call failed: {}", e.getMessage()));
            // 절대 e.getCause() 또는 req.getApiKey() 같은 시크릿 노출 X
    }
}
```

> **절대 금지:**
> - `log.info("Calling Anthropic with key={}", apiKey)` — 시크릿 로그 노출
> - `req.toString()` 로그 — 사용자 입력 (PII 가능) 노출
> - 응답 본문 전체 로그 — 토큰 비용 + 정보 누출

### 3.4 timeout·재시도 정책

| 상황 | 정책 |
| --- | --- |
| Anthropic 응답 < 4분 | 정상 (대부분 30초 이내) |
| Anthropic 응답 > 4분 | timeout → SSE `error` 이벤트 (code: LLM_TIMEOUT) → turn 폐기 (DB save X) |
| Anthropic 5xx (rate limit / 502) | 재시도 X — 즉시 SSE `error` (code: LLM_UNAVAILABLE). 사용자가 수동 재시도 |
| Anthropic 401 (API key 만료) | 재시도 X — 즉시 SSE `error` (code: LLM_AUTH_FAILED) + Slack alert (운영자 즉시 대응) |
| Anthropic 429 (per-minute limit) | 재시도 X — 즉시 사용자에게 SSE `error` (code: RATE_LIMITED) + 60초 후 안내 |

자동 재시도 X 이유: SSE 스트림은 부분 응답 가능 (token N개 이미 발행됨). 재시도 시 사용자가 다른 응답 받으면 혼란. 사용자에게 명시적으로 재시도 buttn 노출 (08번 § 12.3).

### 3.5 비용 알림 임계 (§ 10·§ 12와 정합)

| 임계 | alert 채널 | 동작 |
| --- | --- | --- |
| 1일 비용 USD 5 초과 | Slack `#claude알람채널` (강태오 owner) | 정보 안내 |
| 1일 비용 USD 20 초과 | 동일 + 강상민 직접 멘션 | 즉시 검토 — Anthropic key rotate 가능성 |

| API key 401 1회 발생 | 동일 + 강상민 + 강태오 멘션 | 즉시 key rotate (05번 § 4) |

```
당신은 한국 개신교 큐티 코칭 도우미입니다. 사용자가 오늘 무엇을 결심하고 행동할지 스스로 정하도록 돕는 역할입니다.

## 현재 단계: D. 결단 (Decision)

이 단계의 목표는 C에서 사용자가 발견한 적용을 1개의 구체적 행동·결심으로 좁히는 것입니다. LLM은 결단을 제안하지 않고, 사용자가 자신의 결단을 분명히 말로 정리하도록 돕습니다.

## 본문
{passage_book} {passage_chapter}장 {passage_verse}절

## C 단계에서 사용자가 도출한 적용 (BFF가 자동 요약)
{step_c_summary}

## 좋은 결단의 4가지 조건 (SMART의 큐티 변형)
1. 구체적 (Specific) — "더 사랑하겠다" X / "오늘 저녁에 어머니께 안부 전화 1통 하겠다" OK
2. 측정 가능 (Measurable) — "기도하겠다" X / "오늘 밤 10분 기도하겠다" OK
3. 행동 가능 (Actionable) — 마음·생각만 X, 실제 행동 포함
4. 시간 명시 (Time-bound) — "언젠가" X / "오늘 또는 내일" 명시

## 행동 규칙
- 사용자가 결단을 말하면 위 4가지 조건으로 부드럽게 점검 질문 1개:
  - 구체성 부족 → "더 구체적으로 어떻게 하시겠어요?"
  - 측정 X → "어느 정도 시간 / 횟수로 하시겠어요?"
  - 행동 X (마음만) → "이 결심을 어떤 행동으로 표현하시겠어요?"
  - 시간 X → "오늘이나 내일 중 언제 하시겠어요?"
- 결단이 4가지 조건 충족하면 → "좋은 결단이네요. 이 결단을 묵상 노트에 기록할게요. 큐티를 마칩니다."로 정리.
- 1~2 turn 안에 결단 정리 (D 단계는 길게 끌지 않음)

## 절대 금지
- LLM이 결단 제안 ("이렇게 해보세요" X — 사용자가 정하도록)
- 사용자 결단을 평가 ("그건 부족해요" X)
- 정통 개신교 범위 밖 결단 (이단적·신비적 결단)
- 사용자에게 부담스러운 결단 강요 (작은 결단도 OK)

## 톤
- 한국어, 존댓말
- 1~3문장 짧게
- 격려하고 정리하는 어조
```

## 5. RAG 컨텍스트 표준 (ChromaDB)

### 5.1 ChromaDB 구성

| 항목 | 값 |
| --- | --- |
| 버전 | ChromaDB 0.5.x (2026-05 stable) |
| 배포 | K8s StatefulSet 1 replica + PVC 5Gi (06번 § 5) |
| Collection | 단일 collection `qtai_corpus` (v1.0). v1.1에 단계별 분리 검토 |
| 임베딩 모델 | **`sentence-transformers/paraphrase-multilingual-mpnet-base-v2`** (768 dim) — 한국어 + 영어 양호 |
| 거리 metric | cosine |
| Index type | HNSW (default) |
| 인덱스 파라미터 | M=16, ef_construction=200, ef_search=100 |

> **W0 Lock-in:** 임베딩 모델 변경 시 전체 재인덱싱 필수. 기존 768차원 vector와 다른 모델은 호환 X. ADR + 강태오 페어 검토.

### 5.4 검색 쿼리 구성

```python
# ai-service에서 호출 (Spring Boot → Python FastAPI 사이드카 또는 직접 호출)
def search_rag(book_code: str, chapter: int, verse: int, step: str, user_query: str, k: int):
    # 1. 본문 텍스트 + 사용자 입력 결합
    passage_text = bible_service.get_verse(book_code, chapter, verse)
    query = f"{book_code} {chapter}:{verse} {passage_text} {user_query}"

    # 2. 임베딩 (paraphrase-multilingual-mpnet)
    query_embedding = embedder.embed(query)

    # 3. ChromaDB 검색 (where 절로 book_code 필터링)
    results = collection.query(
        query_embeddings=[query_embedding],
        n_results=k,
        where={
            "$and": [
                {"book_codes": {"$contains": book_code}},
                {"source_type": {"$in": strategy[step]["filter"]["source_type"]}}
            ]
        }
    )

    # 4. (B 단계만) reranker 적용
    if strategy[step].get("rerank"):
        results = rerank(query, results)[:strategy[step]["rerank_top_n"]]

    return results
```

```python

    {"role": "system", "content": system_prompt_a},  # RAG context 없음

```

A·D 단계에서 system prompt는 본문만 포함, RAG section은 비어있음 (또는 "## RAG 출처: 이 단계는 RAG를 사용하지 않습니다." 표기).

### 5.7 RAG 출처 인용 강제 (§ 8.2와 정합)

LLM이 응답에서 RAG context의 docId를 인용하지 않으면 **응답 검증 실패** → SSE `error` (code: RAG_NOT_CITED) → turn 폐기 + 재시도 1회.

검증 룰:
- B 단계: 응답에 `[doc-NNN]` 형식 인용 1개 이상 필수
- C 단계: 응답에 `[doc-NNN]` 형식 인용 1개 이상 권장 (필수 X — 적용은 사용자 삶 위주)
- A·D 단계: RAG 사용 X — 인용 없어야 함

## 6. 신학 가드레일 — 정통 범위·회피 영역

### 6.1 정통 신학 범위 (한국 개신교 주류)

QT-AI는 **한국 개신교 주류 (장로교·감리교·침례교·성결교 등)** 의 공통 분모인 **사도신경·니케아 신경** 수준의 정통 신학을 따른다. 분파 간 차이는 신중하게 회피.

**긍정적 범위 (LLM이 다뤄도 되는 것):**

| 영역 | 내용 |
| --- | --- |
| 삼위일체 | 성부·성자·성령의 한 하나님 (사도신경·니케아 신경 표준) |
| 그리스도론 | 예수 그리스도는 완전한 하나님이며 완전한 사람 (칼케돈 신조) |
| 구원론 | 예수의 십자가 죽음과 부활을 통한 구원, 믿음으로 의롭다 함 (이신칭의) |
| 성경관 | 성경 66권은 하나님의 계시·정경 (정경의 권위) |
| 종말론 | 예수의 재림, 부활, 심판, 새 하늘과 새 땅 |
| 성례 | 세례·성찬 (분파 간 차이 신중하게 — § 6.3 회피) |

### 6.2 회피 영역 (LLM이 다루지 않는 것)

다음 영역은 시스템 프롬프트에서 명시적으로 금지:

### 6.4 자해·자살 멘션 즉시 대응 (가장 중요)

**최우선 가드레일:** 사용자 입력에 자살·자해 키워드 감지 시 정상 LLM 흐름 **즉시 중단**. 코칭 도우미가 위기 개입 X.

흐름:
1. 입력 sanitize 단계 (§ 7.1)에서 regex `자살|자해|죽고 싶|더 살 이유 없|목매|투신` 감지
2. AI Service: SSE `error` 이벤트 발행 (code: `SELF_HARM_DETECTED`, message: 한국어 안내 메시지 + 자살예방상담전화 1393)
3. Flutter: 큰 알림 dialog로 표시 + 큐티 세션 abandon (08번 § 12.5)
4. AI Service: AI_SESSIONS.status = ABANDONED + reason = SELF_HARM_REPORTED
5. Slack alert (`#claude알람채널`) — 강상민·강태오 멘션 (자동 알림은 사용자 명시 요청 후만 — 메모리 룰)
6. 사용자 user_id 정보 X (PII), 단순히 "자해·자살 키워드 감지 시각" 만 alert에 포함

> **윤리 원칙:** LLM은 위기 개입 자격 없음. 즉시 전문 상담 안내 + 인간 운영자에게 알림이 표준. 1393 (한국 자살예방상담전화)·129 (보건복지콜센터) 등 공공 번호 우선.

### 6.5 LLM-as-judge 정통성 검증 (응답 출력 후)

각 LLM 응답 완료 후 별도 짧은 LLM 호출로 정통성 점검:

```
시스템 프롬프트 (judge):
"다음 응답이 한국 개신교 주류 정통 신학 범위에 속하는지 평가해주세요. 
범위 내면 'OK', 범위 밖이면 'OUT_OF_SCOPE_<카테고리>' 반환.
카테고리: HERESY | DENOMINATION_CONFLICT | POLITICS | PROPHECY | MYSTICISM | OTHER

응답: {llm_response}"

판정: OK or OUT_OF_SCOPE_*
```

비용 절감을 위해 judge는 **claude-haiku-4-5** 사용 (Sonnet의 1/10 비용). 응답 시간 ~1초.

**결과 처리:**
- OK → 정상 SSE 발행 + AI_TURNS 저장
- OUT_OF_SCOPE_* → SSE `error` (code: THEOLOGY_GUARDRAIL_VIOLATION) + turn 폐기 + alert

> **W3 평가:** theology-set 50건으로 judge 정확도 평가 (§ 11.3). false positive (정통인데 OUT 판정) > 5% 시 judge 프롬프트 튜닝.

## 7. 프롬프트 인젝션 방어 (3 계층)

### 7.1 계층 1 — 입력 Sanitize

사용자 입력은 LLM에 도달하기 전에 다음 단계 통과:

| 검사 | 동작 |
| --- | --- |
| 길이 제한 | max 500자 (한국어 기준) — 초과 시 400 BAD_REQUEST |
| 인코딩 검증 | UTF-8 정상 + Unicode 정규화 (NFC) — hex injection 차단 |
| 자살·자해 키워드 | regex 감지 → 즉시 § 6.4 흐름 |
| 인젝션 패턴 키워드 | 다음 패턴 감지 시 LLM 안 거치고 바로 `PROMPT_INJECTION_DETECTED` 응답 |
| HTML/SQL 패턴 | `<script>`·`<iframe>`·`SELECT *` 등은 입력에 의미 없음 — strip 또는 거절 |
| Anthropic API key 패턴 | `sk-ant-*` 패턴 — 의심스러운 입력. alert + 거절 |

**인젝션 패턴 키워드 매트릭스:**

```yaml
# ai-service/src/main/resources/safety/injection_patterns_v1.yaml
patterns:
  - name: "ignore_previous"
    regex: "(?i)(이전|위)\\s*(지시|명령|prompt|instruction)\\s*(을|를)?\\s*(무시|ignore|forget)"
    action: "block"
  - name: "system_prompt_leak"
    regex: "(?i)(system\\s*prompt|시스템\\s*프롬프트|당신의\\s*역할)\\s*(을|를)?\\s*(보여|반복|출력)"
    action: "block"
  - name: "role_override"
    regex: "(?i)(당신은\\s*이제)|(now\\s*you\\s*are)|(act\\s*as)"
    action: "block"
  - name: "secret_exfil"
    regex: "(?i)(api\\s*key|secret|password|token)\\s*(을|를)?\\s*(보여|알려|출력)"
    action: "block"
  - name: "jailbreak_classic"
    regex: "(?i)(DAN\\s*mode)|(developer\\s*mode)|(unrestricted\\s*mode)"
    action: "block"
```

탐지 시 응답: `{ code: "PROMPT_INJECTION_DETECTED", message: "입력에 허용되지 않는 내용이 포함되어 있습니다." }` (08번 § 6.5와 정합).

### 7.2 계층 2 — Role 분리 + Delimiter

LLM 호출 시 system role과 user role을 명확히 분리. 사용자 입력은 절대 system prompt에 직접 삽입 X.

```python
# ❌ 안티 패턴 — 사용자 입력이 system prompt에 들어감
system_prompt = f"당신은 큐티 도우미입니다. 사용자 질문: {user_input}"

# ✅ 표준 — system과 user 분리
messages = [
    {"role": "system", "content": SYSTEM_PROMPT_A},  # 고정 박제
    *conversation_history,
    {"role": "user", "content": user_input}          # 입력 sanitize 후 그대로
]
```

**RAG context도 별도 처리** — user_message 안에 X, system prompt 안의 명확한 delimiter 안에:

```
[SYSTEM PROMPT 안]
...
## RAG 출처 (다음은 신뢰할 수 있는 사전 검증된 자료입니다 — 사용자 입력 X)
<rag_context>
[doc-001] ...
[doc-002] ...
</rag_context>
...
```

`<rag_context>` 같은 명시적 delimiter는 LLM이 "이 부분은 검증된 자료"로 인식하도록 도움.

### 7.3 계층 3 — 출력 검증 (§ 8과 정합)

LLM 응답에 다음 패턴이 있으면 차단:

| 패턴 | 의미 | 동작 |
| --- | --- | --- |
| Anthropic API key 패턴 (`sk-ant-*`) | 시스템 프롬프트 누출 가능성 | turn 폐기 + alert |
| 시스템 프롬프트 일부 (정의된 문장) | 누출 | turn 폐기 + alert |
| `Authorization: Bearer ` | JWT 누출 | turn 폐기 + alert |
| 다른 사용자 PII (다른 user_id 정보) | 데이터 격리 위반 | turn 폐기 + alert |

```python
def validate_output(response_text: str) -> bool:
    # API key 패턴
    if re.search(r"sk-ant-[a-zA-Z0-9_-]+", response_text):
        return False
    # JWT 패턴 (eyJ... 의 base64 헤더)
    if re.search(r"eyJ[a-zA-Z0-9_-]+\.eyJ[a-zA-Z0-9_-]+\.[a-zA-Z0-9_-]+", response_text):
        return False
    # Authorization 헤더 패턴
    if "Authorization: Bearer" in response_text:
        return False
    return True
```

검증 실패 시 turn 폐기 + Slack alert (강상민·강태오 멘션) + 다음 turn에서 system prompt 강화 검토.

## 8. 응답 검증 — 출력 가드레일·출처 인용 강제

### 8.1 검증 5종

LLM 응답 완료 후 (또는 token 단위 부분 검증) 5가지 검증:

| # | 검증 | 단계 | 동작 |
| --- | --- | --- | --- |
| 1 | 인젝션 누출 패턴 (§ 7.3) | 모든 단계 | 발견 시 turn 폐기 + alert |
| 2 | 환각 성경 구절 (존재하지 않는 책·장·절) | 모든 단계 | 발견 시 turn 폐기 + 재시도 1회 |
| 3 | RAG 출처 인용 (§ 5.7) | B 필수, C 권장 | B 단계 인용 부재 시 turn 폐기 + 재시도 1회 |
| 4 | 환각 docId (RAG context 안에 없는 docId) | B·C 단계 | 발견 시 turn 폐기 + 재시도 1회 |
| 5 | 신학 정통성 (LLM-as-judge, § 6.5) | 모든 단계 | OUT_OF_SCOPE 시 turn 폐기 + alert |

### 8.2 환각 성경 구절 차단

LLM이 응답에 인용한 모든 성경 구절을 정규식으로 추출 → BOOKS 테이블의 `book_code`·`abbr_kr` 매칭 + KR_BIBLE의 chapter·verse 존재 검증.

```python
import re
from bible_service import bible_repository

# "창세기 1장 1절", "요한복음 3:16", "롬 8:28" 같은 패턴
VERSE_PATTERN = re.compile(
    r"(?P<book>[가-힣]+|[A-Z]{3})\s*(?P<ch>\d+)(?:장|:)\s*(?P<v>\d+)(?:절)?"
)

def validate_verse_citations(response_text: str) -> tuple[bool, list[str]]:
    invalid_citations = []
    for match in VERSE_PATTERN.finditer(response_text):
        book_query = match.group("book")
        chapter = int(match.group("ch"))
        verse = int(match.group("v"))

        # bible_repository — book_code or abbr_kr 매칭
        book = bible_repository.find_book(book_query)
        if not book:
            invalid_citations.append(f"{book_query} (책 없음)")
            continue
        # 해당 장·절이 BOOKS·KR_BIBLE에 존재하는지
        if not bible_repository.verse_exists(book.code, chapter, verse):
            invalid_citations.append(f"{book.kr_name} {chapter}:{verse} (절 없음)")

    return len(invalid_citations) == 0, invalid_citations
```

> **W1 Lock-in:** 본 검증은 AI Service ↔ Bible Service 간 호출. 강상민 ↔ 김태혁 페어 합의 (Bible Service에 `verse_exists(book_code, chapter, verse)` API 추가 — 04번 OpenAPI 정합).

### 8.3 RAG 출처 인용 검증

```python
import re

DOC_ID_PATTERN = re.compile(r"\[doc-\d{3}\]")

def validate_rag_citations(response_text: str, rag_context_doc_ids: list[str], step: str) -> tuple[bool, str]:
    cited = set(DOC_ID_PATTERN.findall(response_text))
    valid = set(f"[{doc_id}]" for doc_id in rag_context_doc_ids)

    # B 단계는 1개 이상 인용 필수
    if step == "B" and len(cited) == 0:
        return False, "RAG_NOT_CITED_IN_STEP_B"

    # 환각 docId
    invalid = cited - valid
    if invalid:
        return False, f"INVALID_DOC_ID: {invalid}"

    return True, ""
```

### 8.4 LLM-as-judge 통합 (§ 6.5)

응답 완료 후 별도 Anthropic Haiku 호출로 정통성 점검. judge 결과는 sync (응답 발행 전 차단).

비용 계산: judge 호출 1회 = ~200 token in + ~10 token out → USD 0.0005/turn. Sonnet 호출의 약 1/40.

### 8.5 검증 실패 시 turn 폐기 흐름

```
LLM 응답 완료 →
  검증 1 (인젝션 누출) FAIL → turn 폐기 + alert (sev: high)
  검증 2 (환각 구절) FAIL → turn 폐기 + 재시도 1회 → 재시도도 FAIL이면 사용자에게 SSE error
  검증 3 (RAG 인용 부재) FAIL → turn 폐기 + 재시도 1회
  검증 4 (환각 docId) FAIL → turn 폐기 + 재시도 1회
  검증 5 (LLM-as-judge OUT_OF_SCOPE) FAIL → turn 폐기 + alert (sev: medium)
  
  모든 검증 PASS → SSE turn_completed 발행 + AI_TURNS DB INSERT
```

재시도 1회는 자동, 다음 회는 사용자에게 노출 (08번 § 12.3 — 자동 재시도는 인증·idempotent만).

## 9. SSE 스트리밍 정책 (04번 § 6.3 정합)

### 9.1 SSE 4 이벤트 종류 (재정합)

| event | data | 의미 | Flutter 처리 |
| --- | --- | --- | --- |
| `token` | `{"text": "..."}` | LLM 토큰 1개 (Anthropic stream으로부터) | UI에 누적 표시 (08번 § 8.4) |
| `turn_completed` | `{"turnId": int, "ragSources": [{"docId": str, "relevance": float}]}` | 한 turn 완료 + 검증 통과 | DB 저장된 turnId·출처 표시 |
| `error` | `{"code": str, "message": str}` | 오류 (검증 실패·timeout 등) | SnackBar + 재시도 옵션 |
| `end` | `[DONE]` 또는 빈 객체 | 스트림 정상 종료 | 입력창 활성화 |

### 9.2 token 발행 표준

- Anthropic stream의 `content_block_delta` 이벤트에서 텍스트 추출 → 즉시 SSE token 발행
- 버퍼링 X (사용자 즉시성)
- 부분 검증 (§ 7.3 인젝션 누출 패턴) 은 token 단위로 — 매번 regex 적용
- 출력 검증 (§ 8.2~§ 8.5) 은 응답 완료 후 — turn_completed 발행 직전

```java
// ai-service/src/main/java/com/qtai/ai/sse/AiSseHandler.java
public Flux<ServerSentEvent<String>> handleTurn(TurnRequest req) {
    StringBuilder fullResponse = new StringBuilder();

    return anthropicClient.streamMessage(buildAnthropicReq(req))
        .flatMap(event -> switch (event.type()) {
            case "content_block_delta" -> {
                String token = event.delta().text();
                fullResponse.append(token);
                // 부분 검증 (인젝션 누출 패턴)
                if (containsApiKeyPattern(fullResponse.toString())) {
                    yield Flux.just(errorEvent("INJECTION_LEAK_DETECTED"));
                }
                yield Flux.just(tokenEvent(token));
            }
            case "message_stop" -> {
                // 응답 완료 — 전체 검증
                String response = fullResponse.toString();
                ValidationResult result = validateResponse(response, req.step(), req.ragContext());
                if (!result.isValid()) {
                    yield Flux.just(errorEvent(result.code()));
                }
                // judge 호출
                JudgeResult judge = judgeResponse(response);
                if (judge.outOfScope()) {
                    yield Flux.just(errorEvent("THEOLOGY_GUARDRAIL_VIOLATION"));
                }
                // turn DB 저장
                Turn turn = turnRepository.save(req.sessionId(), response, req.step(), result.ragSources());
                yield Flux.just(
                    turnCompletedEvent(turn.id(), result.ragSources()),
                    endEvent()
                );
            }
            default -> Flux.empty();
        });
}
```

### 9.3 SSE Gateway 우회 (04번 § 9.6 정합)

Spring Cloud Gateway는 기본적으로 buffering — SSE 스트림 끊김. AI Service의 `/ai/sessions/{id}/turns` path는 Gateway 설정에서 buffering 우회.

```yaml
# gateway/src/main/resources/application.yml
spring:
  cloud:
    gateway:
      routes:
        - id: ai-service-sse
          uri: lb://ai-service
          predicates:
            - Path=/api/v1/ai/sessions/*/turns
            - Method=POST
          filters:
            - StripPrefix=2
            - name: NoBuffering    # 커스텀 filter (강태오 owner)
```

> **김지민 ↔ 강상민 페어 합의 (W1):** Gateway buffering 우회 검증은 W1 5/22 통합 테스트 — Anthropic mock + Flutter SSE 클라이언트 1턴 E2E.

### 9.4 turn 부분 실패 처리 (1차 트랜잭션 사고 매핑)

LLM 응답 중 timeout 발생 시:
- 이미 발행된 token N개는 사용자가 본 상태
- AI_TURNS DB에는 **저장 X** (불완전 응답 저장하면 다음 turn에 컨텍스트 깨짐)
- SSE `error` (code: LLM_TIMEOUT) 발행 → Flutter가 SnackBar
- Kafka 발행 X (turn 미완료)

> **트랜잭션 경계:** turn DB save = SSE `turn_completed` 발행 직전. 이 둘이 atomic이어야 함. 중간 실패 시 turn 자체가 폐기된 것으로 처리 (01번 § 9.3 트랜잭션 사고 매핑).

### 9.5 단계 전환 race 방지 (1차 race 사고 매핑)

A→B→C→D 단계 전환은 **AI Service 내부 자동 판단** — `POST /api/v1/ai/sessions/{id}/turns` 호출 시 답변을 분석하여 `current_step` 갱신. 이 처리 시 **AI_SESSIONS row를 SELECT FOR UPDATE** + current_step 조건 확인:

```sql
-- 단계 전환 트랜잭션
BEGIN;
SELECT current_step FROM AI_SESSIONS WHERE id = :sessionId FOR UPDATE;
-- 검증: A→B만 허용 (역행 X·건너뛰기 X)
UPDATE AI_SESSIONS SET current_step = 'B' WHERE id = :sessionId AND current_step = 'A';
-- 영향 받은 row 0개면 INVALID_STEP_TRANSITION 응답
COMMIT;
```

이 패턴은 02번 § 7 + 04번 § 7.4 에 박제 (Journal status 전이와 동일 패턴).

| 응답 본문 로그 | 토큰·메타만, 본문 X (PII + 비용) |


### 10.2 비용 보호 — 다층 방어

| 계층 | 한도 | 위치 |
| --- | --- | --- |
| Gateway rate | AI 호출 60/min/user | Spring Cloud Gateway (강태오) |
| AI Service per-session | 1 세션 max 20 turn | AI Service in-memory cache |
| AI Service per-user-day | 1 사용자 max 5 세션 / 24h | DB query (AI_SESSIONS count) |
| Anthropic max_tokens | 800~1200 / turn (§ 3.2) | API 파라미터 |
| Anthropic timeout | 4분 / turn (§ 3.4) | WebClient timeout |
| 1일 비용 alert | USD 5·20 (§ 3.5) | Prometheus + Slack |

각 계층 위반 시:
- Gateway: 429 RATE_LIMITED
- AI Service per-session: 409 SESSION_TURN_LIMIT_EXCEEDED → 사용자에게 "세션이 너무 길어졌습니다. 새 세션을 시작해주세요."
- AI Service per-user-day: 429 DAILY_SESSION_LIMIT → 사용자에게 "오늘 세션 한도 초과"
- max_tokens 도달: 응답이 잘림 — 정상 동작 (사용자에게 자연스럽게 표시)
- timeout: SSE error (LLM_TIMEOUT) — § 9.4

### 10.3 timeout 정책 (재정합)

| 영역 | timeout | 근거 |
| --- | --- | --- |
| Anthropic API 호출 (전체 SSE) | 4분 | LLM 일부 응답 30s+ 가능, 4분 안에 끝나면 사용자 인내 한계 |
| Gateway → AI Service | 5분 | Anthropic timeout(4분) + 네트워크·검증 여유 |
| Flutter → Gateway | 5분 | Gateway timeout과 동일 |
| LLM-as-judge 호출 | 30초 | Haiku 짧은 호출 |
| ChromaDB 쿼리 | 5초 | 5초 초과는 ChromaDB 문제 |
| Bible Service `verse_exists` 호출 | 2초 | 단순 PK lookup |

### 10.4 동시 호출 제한 — Anthropic API 부하 보호

Anthropic 계정 tier에 따라 RPM (requests per minute) 한도 존재. v1.0 시연 환경에서는 ~50 RPM.

AI Service에서 in-process semaphore로 동시 Anthropic 호출 max 10:

```java
@Component
public class AnthropicConcurrencyLimiter {
    private final Semaphore semaphore = new Semaphore(10);

    public <T> Mono<T> withLimit(Supplier<Mono<T>> call) {
        return Mono.using(
            () -> { semaphore.acquire(); return semaphore; },
            sem -> call.get(),
            Semaphore::release
        );
    }
}
```

- semaphore 대기 시간 > 10초 시 SSE error (code: LLM_QUEUED_TIMEOUT) → 사용자 재시도 안내
- 동시 호출 metric: `ai_concurrent_calls_active`

### 10.5 비용 모니터링 (§ 3.5와 통합)

```
Anthropic 응답에는 `usage.input_tokens` + `usage.output_tokens` 포함.

매 호출마다 metric 발행:
  ai_anthropic_input_tokens_total{step}
  ai_anthropic_output_tokens_total{step}
  ai_anthropic_cost_usd_total{step} (token × 가격 계산)

Grafana 대시보드:
  - 시간당 비용
  - 1일 누적 비용 (00시 reset)
  - step별 비용 분포
```

Slack alert (§ 3.5) — 메모리 룰: 사용자 명시 요청 후만 자동 전송.

### 15.1 시연 시나리오 (6/17 발표 - 03번 § 1과 정합)

**데모 본문:** 창세기 1장 1절 (가장 친숙·짧음·정통 해석 풍부)

```
1. Flutter 앱 실행 (Android emulator)
2. 로그인 (5초)
3. 대시보드 → 오늘의 구절 카드 (창세기 1:1) 클릭 → 입체 묵상 화면 (5초)

   사용자: "하나님이요"
   AI: "맞습니다! 그 분이 무엇을 하시는지도 본문에서 한 번 찾아보시겠어요?" (3초)
   AI: "관찰을 충분히 다뤘네요. B 해석 단계로 넘어가시겠어요?"

6. B 단계 (해석) — 2 turn 시연
   사용자: "넘어가요"
   사용자: "'창조하시니라'는 무엇을 의미하나요?"
   AI: "히브리어 'bara'는 '무에서 유를 만들다'의 의미입니다 [doc-001]. 이는 ..." (10초)

7. C 단계 (적용) — 1 turn 시연
   사용자: "이 의미가 제 삶에 어떻게 적용되나요?"
   AI: "오늘 하루 중 어떤 부분에서 '하나님이 시작하셨다'를 떠올릴 수 있을까요?" (5초)

8. D 단계 (결단) — 1 turn 시연
   사용자: "오늘 저녁 기도 시간에 창조하신 하나님께 감사를 표현하겠어요"
   AI: "구체적인 시간이 정해져 있어 좋네요! 이 결단을 묵상 노트에 기록할게요. 큐티를 마칩니다."

9. STOMP 알림 도착 (3초): "묵상 노트가 자동으로 생성되었습니다"
```

### 15.2 안전 fallback — 시연 중 LLM 장애 시

**시나리오 1: Anthropic API 장애 (5xx)**
- AI Service가 SSE error (LLM_UNAVAILABLE) 발행
- Flutter SnackBar: "AI 응답에 일시적 문제가 있습니다"
- **Lead가 사전 녹화 영상 재생** — 같은 시나리오의 정상 데모 영상

**시나리오 2: 검증 실패 반복 (환각·인젝션)**
- AI Service가 SSE error (VALIDATION_FAILED) 발행
- Flutter SnackBar: "응답을 다시 생성하겠습니다"
- 1회 자동 재시도 → 그래도 실패 시 Lead가 다음 시나리오로 진행

**시나리오 3: 시연 중 비용 폭주 (USD 1+ 1세션)**
- AI Service per-session turn 한도 (20)에 의해 자동 차단
- Flutter: "세션 한도에 도달했습니다. 새 세션을 시작해주세요"

**시나리오 4: 자해 키워드 demo 시 (절대 시연하지 말 것)**
- 리허설 + 발표 모두 자해 키워드 입력 X
- 발표 자료에는 "안전 가드레일이 있다" 설명만 + injection-set 통계만 표시

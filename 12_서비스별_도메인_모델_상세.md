## 1. 클린 아키텍처 계층 원칙

### 1.1 4 계층 규칙

```
Presentation (Controller, DTO)
    ↓ (의존)
Application (UseCase, Service)
    ↓
Domain (Entity, Repository interface, Domain Exception)
    ↑ (구현)
Infrastructure (JPA, Redis, Kafka, REST Client)
```

**핵심 규칙:**

| 계층 | 허용 의존 | 금지 의존 |
| --- | --- | --- |
| Domain | 없음 (순수 Kotlin) | Spring, JPA, Kafka 등 모든 프레임워크 |
| Application | Domain | Spring Web, JPA 직접 |
| Presentation | Application | Domain Entity 직접 노출 |
| Infrastructure | Domain (interface 구현) | Application 직접 |

**1차 사고 재발 방지:**
```kotlin
// ❌ 1차 사고 패턴 — Controller에 비즈니스 로직
@PostMapping("/login")
fun login(@RequestBody req: LoginRequest): ResponseEntity<TokenResponse> {
    val user = userRepository.findByEmail(req.email) ?: throw Exception("not found")
    if (!passwordEncoder.matches(req.password, user.passwordHash)) throw Exception("wrong")
    val token = jwtService.generateToken(user.id)
    return ResponseEntity.ok(TokenResponse(token))
}

// ✅ 2차 표준 — UseCase로 위임
@PostMapping("/login")
fun login(@RequestBody req: LoginRequest): ResponseEntity<TokenResponse> =
    loginUseCase.execute(LoginCommand(req.email, req.password))
        .let { ResponseEntity.ok(it.toResponse()) }
```

### 2.2 RefreshToken 도메인 엔티티

```kotlin
// domain/entity/RefreshToken.java
data class RefreshToken(
    val id: Long,
    val userId: Long,
    val tokenHash: String,        // 평문 저장 X — SHA-256 hash
    val expiresAt: LocalDateTime, // 14일
    val revokedAt: LocalDateTime?,
    val revokeReason: RevokeReason?,
    val isUsed: Boolean,          // Rotation: 사용 시 true + 새 token 발급
    val createdAt: LocalDateTime,
) {
    fun isValid(): Boolean =
        revokedAt == null && !isUsed && expiresAt.isAfter(LocalDateTime.now())

    fun revoke(reason: RevokeReason): RefreshToken = copy(
        revokedAt = LocalDateTime.now(),
        revokeReason = reason,
    )

    fun markUsed(): RefreshToken = copy(isUsed = true)
}

enum class RevokeReason { ROTATION, LOGOUT, REUSE_DETECTED, ADMIN }
```

### 2.4 로그인 비즈니스 규칙

```kotlin
// application/usecase/LoginUseCase.java
class LoginUseCase(
    private val userRepository: UserRepository,
    private val refreshTokenRepository: RefreshTokenRepository,
    private val passwordEncoder: PasswordEncoder,
    private val jwtService: JwtService,
    private val redisLockService: RedisLockService,
) {
    fun execute(command: LoginCommand): TokenPair {
        val user = userRepository.findByEmail(command.email)
            ?: throw UserNotFoundException()

        if (!user.isActive()) throw UserDeactivatedException()

        if (!passwordEncoder.matches(command.password, user.passwordHash))
            throw InvalidCredentialsException()

        // Login rate limit: 1회/5초 (05번 § 3.1)
        val lockKey = "login:rate:${user.id}"
        redisLockService.acquireOrThrow(lockKey, ttl = Duration.ofSeconds(5)) {
            throw LoginRateLimitException()
        }

        val accessToken = jwtService.generateAccessToken(user)
        val refreshToken = jwtService.generateRefreshToken(user)
        refreshTokenRepository.save(refreshToken)

        return TokenPair(accessToken, refreshToken.plainToken)
    }
}
```

## 5. AI Service 도메인 모델

### 5.1 AiSession 도메인 엔티티

```kotlin
// domain/entity/AiSession.java
data class AiSession(
    val id: Long,
    val userId: Long,
    val bookCode: String,
    val chapter: Int,
    val verse: Int,
    val currentStep: SessionStep,
    val status: SessionStatus,
    val createdAt: LocalDateTime,
    val updatedAt: LocalDateTime,
    val abandonReason: AbandonReason?,
) {
    fun canAdvanceTo(nextStep: SessionStep): Boolean {
        return when (currentStep) {
            SessionStep.A -> nextStep == SessionStep.B
            SessionStep.B -> nextStep == SessionStep.C
            SessionStep.C -> nextStep == SessionStep.D
            SessionStep.D -> false  // D가 마지막
        }
    }

    fun isActive(): Boolean = status == SessionStatus.ACTIVE

    fun advance(nextStep: SessionStep): AiSession {
        require(canAdvanceTo(nextStep)) {
            "Cannot advance from $currentStep to $nextStep"
        }
        return copy(currentStep = nextStep)
    }

    fun complete(): AiSession = copy(status = SessionStatus.COMPLETED)

    fun abandon(reason: AbandonReason): AiSession = copy(
        status = SessionStatus.ABANDONED,
        abandonReason = reason,
    )
}

enum class SessionStep { A, B, C, D }
enum class SessionStatus { ACTIVE, COMPLETED, ABANDONED }
enum class AbandonReason { USER_REQUEST, SELF_HARM_DETECTED, TIMEOUT, ERROR }
```

## 6. BFF Aggregator 도메인 모델

### 6.1 BFF의 역할

BFF(Backend For Frontend)는 **도메인 로직 없음** — 오직 집계·변환·프록시:

```
Flutter → BFF
  Gateway → BFF       (검증된 X-User-Id 전달)
  BFF → 여러 Service  (Aggregation)
  BFF → Flutter       (통합 응답)
```

### 6.2 대시보드 집계 UseCase

```kotlin
// application/usecase/GetDashboardUseCase.java
class GetDashboardUseCase(
    private val bibleClient: BibleServiceClient,
    private val journalClient: JournalServiceClient,
    private val aiSessionClient: AiSessionServiceClient,
) {
    fun execute(userId: Long): DashboardData {
        // 병렬 호출 (Kotlin Coroutines)
        return runBlocking {
            val todayVerse = async { bibleClient.getTodayVerse() }
            val recentJournals = async { journalClient.listRecent(userId, limit = 3) }
            val activeSession = async { aiSessionClient.getActive(userId) }

            DashboardData(
                todayVerse = todayVerse.await(),
                recentJournals = recentJournals.await(),
                activeSession = activeSession.await(),
            )
        }
    }
}
```

### 6.3 STOMP 알림 집계

BFF는 `notification.requested` Kafka 토픽을 consume하여 해당 user의 STOMP 세션으로 전달. 비즈니스 로직 없음 (ADR-0012 — Notification stateless).

```kotlin
// infrastructure/kafka/NotificationConsumer.java
@KafkaListener(topics = ["notification.requested"])
fun onNotification(event: NotificationRequestedEvent, ack: Acknowledgment) {
    val session = stompSessionRegistry.getSession(event.userId)
    session?.send("/user/queue/notifications", event.toStompMessage())
    ack.acknowledge()
}
```

```kotlin
@Component
@Transactional  // ← 모든 쓰기 UseCase에 필수 (1차 사고 재발 방지)
class CreateJournalUseCase(
    private val journalRepository: JournalRepository,
    private val eventPublisher: DomainEventPublisher,
) : UseCase<CreateJournalCommand, Journal> {

    override fun execute(input: CreateJournalCommand): Journal {
        // 1. 도메인 유효성 검증
        require(input.content.isNotBlank()) { "Content must not be blank" }

        // 2. 도메인 생성
        val journal = Journal(
            userId = input.userId,
            title = input.title,
            content = input.content,
            // ...
        )

        // 3. 저장
        val saved = journalRepository.save(journal)

        // 4. 이벤트 발행 (AFTER_COMMIT — 03번 § 2.3)
        eventPublisher.publish(JournalCreatedEvent(saved.id, saved.userId))

        return saved
    }
}
```

```kotlin
class JournalQueryService(
    private val journalRepository: JournalRepository,
) {
    fun findById(id: Long, userId: Long): Journal {
        val journal = journalRepository.findById(id)
            ?: throw JournalNotFoundException(id)

        return journal
    }
}
```

## 10. BaseEntity 표준 + 소스 복사 원칙

### 10.1 BaseEntity (ADR-0009 — 소스 복사)

MSA에서 공통 BaseEntity를 공유 라이브러리로 만들면 **배포 결합도 발생** (한 서비스 변경이 전체 빌드 영향). ADR-0009: 소스 코드 직접 복사 방식 채택.

```kotlin
// 각 서비스의 domain/entity/BaseEntity.java (동일 코드 복사)
@MappedSuperclass
abstract class BaseEntity {
    @CreatedDate
    @Column(updatable = false)
    lateinit var createdAt: LocalDateTime

    @LastModifiedDate
    lateinit var updatedAt: LocalDateTime
}
```

**복사 규칙:**
- 서비스 초기 셋업 시 `common/` 의 `BaseEntity.java`를 각 서비스의 domain 패키지에 복사
- 이후 독립적 수정 가능 (서비스별 필드 추가 등)
- `common/` 모듈에는 UseCase interface, ErrorCode, DTO 공통 클래스만

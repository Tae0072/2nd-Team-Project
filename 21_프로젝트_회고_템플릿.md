## 1. 회고 진행 방법

### 1.1 일정

| 시점 | 내용 | 시간 |
| --- | --- | --- |
| **6/17 발표 당일 오후** | 발표 소감 + 즉각 감사 (5분) | 15분 |
| **6/18(목) 오전 10:00** | 본 회고 (팀 KPT + 1차 비교) | 60~90분 |
| **6/18 오후** | 개인 성장 기록 각자 작성 | 30분 (개별) |

### 1.2 KPT 진행 방식

```
1. 포스트잇 (또는 Slack 스레드) 각자 3분 작성
   - Keep: 잘됐고 다음에도 유지할 것
   - Problem: 아쉬웠던 것 / 다음에는 바꿀 것
   - Try: 다음 프로젝트에서 시도할 것

2. 강태오가 분류·그룹핑 (5분)

3. 그룹별 토론 (40분)
   - 가장 많이 나온 Problem 3개를 중심으로
   - 각 Problem에 Try 1개씩 연결

4. 최종 Action Item 도출 (15분)
   - 다음 프로젝트에서 반드시 할 것 3~5개
   - 담당자 지정 (있다면)
```

### 1.3 회고 원칙

```
✅ 사람 비판 X, 시스템·프로세스 비판
✅ "나는 ~느꼈다" 형식으로 감정 표현 OK
✅ 모든 의견 동등하게 (직급·실력 무관)
✅ 발표 결과(점수·평가)로 품질 판단 X — 배움으로 판단
✅ 1차에서 못 했던 것을 2차에서 했다면 명시적으로 칭찬
```

## 3. 1차(HMS) ↔ 2차(QT-AI) 비교

### 3.1 기술적 개선 여부 (회고 당일 ✅/❌/△ 평가)

| 1차 사고 패턴 | 2차 가드레일 | 실제 재발? | 비고 |
| --- | --- | --- | --- |
| Spring Boot 메서드 환각 | § 9.2 grep scan | ✅ / ❌ / △ | |
| @Transactional 누락 | CI Detekt 경고 | ✅ / ❌ / △ | |
| 평문 시크릿 commit | TruffleHog CI 스캔 | ✅ / ❌ / △ | |
| Race condition | Redis Distributed Lock | ✅ / ❌ / △ | |
| 에러 메시지 직접 노출 | ProblemDetail 매핑 | ✅ / ❌ / △ | |
| 운영 가시성 없음 | Grafana·Tempo·Loki | ✅ / ❌ / △ | |
| DB 인덱스 없음 | idx_kr_bible_bcv | ✅ / ❌ / △ | |
| Kafka 멱등성 없음 | UNIQUE + manual ack | ✅ / ❌ / △ | |
| 보안 헤더 미전달 | Gateway X-User-Id | ✅ / ❌ / △ | |
| LLM 환각 | verse_exists 검증 | ✅ / ❌ / △ | |

**평가 기준:**
- ✅ 가드레일이 실제로 효과가 있었음
- ❌ 가드레일을 만들었지만 여전히 유사 문제 발생
- △ 완벽하진 않았지만 1차보다는 나아짐

### 3.2 아키텍처 레벨 비교

| 항목 | 1차 (HMS) | 2차 (QT-AI) | 개선됨? |
| --- | --- | --- | --- |
| 아키텍처 | Spring Boot 모놀리식 | MSA 6 서비스 | △ (복잡도 증가) |
| DB | MySQL 단일 (JPA) | MySQL per service + Redis | ✅ |
| 인증 | 세션 기반 | JWT RS256 + Refresh Rotation | ✅ |
| 프론트엔드 | Mustache SSR + jQuery | Flutter 모바일 | ✅ |
| 실시간 | WebSocket (단순) | SSE + STOMP + Kafka | △ |
| AI 통합 | Claude API (직접) | MSA AI Service + RAG + 가드레일 | ✅ |
| 테스트 | 커버리지 X | CI 게이트 60%+ | ✅ |
| 문서화 | 기본 README | 01~21번 전체 문서 세트 | ✅ |
| 배포 | Docker Compose | K8s (Minikube) + Helm | △ (러닝커브) |

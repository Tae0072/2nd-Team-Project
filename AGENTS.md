## 1. Project Summary

QT-AI is a Quiet Time app for today's Bible passage, easy Korean explanations, personal journals, curated worship songs, anonymous sharing, simulator status, and admin operations.

- **Implementation target:** one `qtai-server` Modular Monolith + Flutter app/admin surface

## 3. Tech Stack

- **Backend:** Java 21, Spring Boot 3.3, Gradle, Spring Modulith, ArchUnit
- **Backend shape:** single `qtai-server`; no independent services
- **DB/cache:** MySQL 8.0, H2 for tests, Caffeine first, Redis if reviewed
- **AI:** DeepSeek OpenAI-compatible client, batch/admin trigger only
- **Frontend:** Flutter app; admin UI by agreed Flutter Web/admin surface
- **CI/deploy:** GitHub Actions, Spectral, JaCoCo, gitleaks, Docker Compose

## 4. Architecture Rules

```text
Flutter App / Admin Web
  -> /api/v1/** external HTTP API
  -> qtai-server
      +-- gatewayauth
      +-- bff
      +-- bible
      |   +-- journal
      |   +-- songs
      +-- ai
      +-- simulator
```

- `journal` and `songs` are `bible` submodules, not top-level domains.
- External APIs are HTTP paths under `/api/v1/**`.
- Internal domain calls use Java Interfaces and DTOs, not HTTP paths.
- Other domains may import only `api/` Port and DTO packages.
- Do not import another domain's Entity, Service, Repository, or infrastructure type.
- Controllers must not call repositories directly.
- User-facing request paths must not call LLM clients.

## 5. Fixed Product Decisions

- QT range is published at **00:00 KST**; our collection batch runs at **04:00 KST**.
- From 00:00 to before 04:00, serve the previously prepared QT cache.
- "Today QT 100%" means passage, explanation entry point, journal entry point, and simulator status are returned.
- It does not mean every passage has an actual simulator clip.
- Simulator status is `READY`, `MISSING`, `FAILED`, or `DISABLED`; enable the button only for `READY`.
- AI content is pre-generated by batch or admin trigger only.
- A/B/C explanation model: A is internal Korean reference, B is public-domain English source, C is verified user-visible Korean explanation.
- Songs are operator-curated metadata. Store user song references only.
- Soft login: first entry, today's QT preview, Bible list/passage, and curated songs can be public.

## 6. Forbidden Work

Do not add or scaffold these, even temporarily:

- User AI Q&A, chatbot, SSE, `/ai/sessions/**`, or real-time user LLM calls
- RAG, ChromaDB, vector DB, Elasticsearch
- Kafka, Kubernetes, Helm for v1
- Church verification screens, buttons, APIs, or DB fields
- AI worship-song recommendations
- Lyrics, audio files, or direct YouTube URL input/storage
- Korean Revised Version, ESV, NIV seed/test/fixture/response data
- Scripture Union or Duranno Bible text storage; collect QT range information only
- Plain secrets in code, YAML, examples, logs, or docs

Use cautious copyright wording: "저작권 리스크를 낮춘다".
For events, describe handler failure logs and reprocessing ability; do not promise perfect delivery.

## 7. Code Rules

- Use `jakarta.*`, not `javax.*`.
- Use `@Transactional` on write use cases and command services; use `readOnly = true` for read paths where appropriate.
- Keep domain models inside their domain. Return DTOs across boundaries.
- Record AI runs with `promptVersion`, `modelName`, `inputHash`, `outputHash`, `status`, and error details.
- Admin APIs require `ADMIN`; batch/AI internal work is recorded as `SYSTEM`.
- Event handlers must log `eventId`, event type, handler name, and error message on failure.
- Logs must not include passwords, tokens, private keys, or sensitive personal data.
- Prefer Korean for user-facing messages, PR explanations, and domain comments.

## 8. Tests And Commands

Run relevant checks before PR:

```bash
./gradlew build
./gradlew test jacocoTestReport
./gradlew jacocoTestCoverageVerification
npx @stoplight/spectral-cli lint apis/*/openapi.yaml --ruleset .spectral.yaml
gitleaks detect --source . --redact --exit-code 1
```

If `flutter-app` exists:

```bash
cd flutter-app
flutter pub get
flutter analyze
flutter test --coverage
```

Required test areas: 00:00/04:00 today QT behavior, simulator statuses, no user-request LLM calls, A/B non-exposure, admin authorization, Bible source metadata, event failure logging, and domain boundary guards.

## 9. PR Rules

- Branch from `dev`; never push directly to `dev` or `master`.
- Use `{type}/{scope}-{short-description}` branch names and Conventional Commits.
- Keep PRs small: preferably 10 files or fewer and 500 changed lines or fewer.
- Feature PRs must reference the relevant F-ID from `07_요구사항_정의서.md`.
- Architecture/API/quality/term changes must link `03`, `04`, `18`, or `23`.
- Do not stage `.gradle`, build outputs, coverage HTML, generated reports, or temporary files.

## 2. 고정 기준

| 구분 | 기준 |
| --- | --- |
| 첫 진입 | 로그인 전에는 로그인 화면으로 진입하고, 로그인 후 별도 홈 화면 없이 Today QT 화면으로 이동한다. |
| 로그인 경계 | Today QT, 성경 본문, 저장, 해설, 찬양, 달력 등 기능 화면은 로그인 후 접근 |
| Today QT 100% | 시뮬레이터 클립이 없어도 상태값을 표시하고 버튼을 비활성화하면 정상 흐름이다. |
| 시뮬레이터 버튼 | `READY`만 활성화, `MISSING`/`FAILED`/`DISABLED`는 비활성화 |
| 금지 UI | AI 자유 챗봇, AI 찬양 추천, 교회 인증 버튼은 MVP 화면에 두지 않는다. |
| 관리자 | 관리자 페이지는 **별도 관리자 웹** (Flutter 앱 아님). 오늘 QT, 승인 해설, 시뮬레이터 상태, 신고, 찬양 큐레이션을 관리 대상으로 둔다. |
| 관리자 분담 | 바이블서버 팀 3명 공동. 본인은 관리자 웹 신고 처리/시뮬레이터 상태 메뉴 주 담당. 이지윤은 QT/해설 메뉴, 이승욱은 공지/찬양 메뉴 주 담당 (§4.9.2 참조) |

## 5. 브랜치·PR·워크플로우 기록 규칙

| 항목 | 김지민 기준 |
| --- | --- |
| 브랜치 생성 | `dev` 최신화 후 `feature/mobile-*`, `feature/admin-*`, `feature/report-*`, `bugfix/mobile-*`, `bugfix/admin-*`, `bugfix/report-*` 형식으로 생성 (관리자 웹 = admin- prefix, Flutter 앱 = mobile- prefix) |
| PR 대상 | 항상 `dev` |
| 자동 PR | 브랜치 push 후 GitHub PR을 생성하면 CI와 자동 리뷰가 실행된다. 자동 머지는 하지 않는다. |
| 작업 폴더 | 별도 구현 GitHub에서 `workspaces/김지민/workflows`, `workspaces/김지민/reports`를 사용 |
| 기록 순서 | 작업 전 workflow 작성 → 작업·검증 → report 작성 → PR 생성 |
| 기준 문서 | 상세 명령과 템플릿은 `개발자별_일정표/00_공통_브랜치_PR_워크플로우_규칙.md`를 따른다. |
